Add a parent pointer to this tree node — Solution

March 2026 · Part B: The Solution

The answer: A

Store all nodes in a flat Vec and use indices instead of pointers. This is the arena pattern.

Parent pointers create a cycle: the parent owns the child, and the child points back to the parent. Rust's ownership model is a tree — one owner per value, no cycles. You can't express "A owns B, and B borrows A" without running into the borrow checker, because mutating the parent would invalidate the child's reference.

The arena pattern sidesteps this entirely. All nodes live in a Vec. Relationships are expressed as indices — plain usize values. No references, no lifetimes, no reference counting. You look up a node's parent by indexing into the same vec. It's the same strategy that ECS game engines, compilers, and graph libraries use in Rust.

This is a fundamental Rust pattern: when your data structure has cycles or shared references, flatten it into an arena and use IDs.

The solution

struct Node {
    value: String,
    parent: Option<usize>,    // index into the arena
    children: Vec<usize>,     // indices into the arena
}

struct Tree {
    nodes: Vec<Node>,
}

impl Tree {
    fn new(root_value: &str) -> Self {
        Tree {
            nodes: vec![Node {
                value: root_value.into(),
                parent: None,
                children: vec![],
            }],
        }
    }

    fn add_child(&mut self, parent: usize, value: &str) -> usize {
        let child_idx = self.nodes.len();
        self.nodes.push(Node {
            value: value.into(),
            parent: Some(parent),
            children: vec![],
        });
        self.nodes[parent].children.push(child_idx);
        child_idx
    }
}

fn main() {
    let mut tree = Tree::new("root");
    let c1 = tree.add_child(0, "child-1");
    let c2 = tree.add_child(0, "child-2");

    // Navigate from child back to parent
    if let Some(parent_idx) = tree.nodes[c1].parent {
        println!("{}'s parent is {}", tree.nodes[c1].value, tree.nodes[parent_idx].value);
    }
}

Why the other options fail

B. Box<Node> for parent — this means each child owns a copy of its parent, which owns copies of all its children, which each own copies of the parent… infinite recursion. It's not a pointer, it's a copy.

C. &Node reference — you can't store a reference to the parent because the parent is being mutated (adding children). The borrow checker won't allow a shared reference to exist while you hold a mutable one, and the lifetimes get impossible to satisfy when the tree structure changes.

D. Rc<RefCell<Node>> everywhere — this creates a reference cycle between parent and child. Both hold strong Rc pointers to each other, so neither's reference count ever reaches zero. Memory leak. You'd need Weak for at least one direction, adding complexity that the arena pattern avoids entirely.

When your data has cycles, flatten it into an arena and use indices as pointers.
