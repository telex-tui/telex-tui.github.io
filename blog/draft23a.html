A settings store that multiple components read and one component updates.

use std::collections::HashMap;

struct Settings {
    values: HashMap<String, String>,
}

impl Settings {
    fn new() -> Self {
        Settings { values: HashMap::new() }
    }

    fn get(&self, key: &str) -> Option<&str> {
        self.values.get(key).map(|s| s.as_str())
    }

    fn set(&mut self, key: &str, value: &str) {
        self.values.insert(key.into(), value.into());
    }
}

fn main() {
    let mut settings = Settings::new();
    settings.set("theme", "dark");
    println!("theme: {:?}", settings.get("theme"));
}

Two threads need to read settings concurrently. A third thread occasionally writes new settings. Choose the right synchronisation primitive.

Your options

  • Use Arc<RwLock<Settings>> — readers don't block each other, writers get exclusive access
  • Use Arc<Mutex<Settings>> — one lock for all access, readers and writers alike
  • Clone the settings for each reader so they have their own independent copy
  • Use Cell<Settings> so values can be updated without a mutable reference

Pick one before reading the solution.
