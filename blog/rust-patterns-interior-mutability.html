Rust Patterns That Matter #1: Interior Mutability

March 2026

Post 1 of 20 in Rust Patterns That Matter.

You have a struct with a HashMap field that you use for caching, and it gets passed around by shared reference. Inside a method that takes &self, you need to insert into that HashMap — and the compiler rejects it. This is the first wall.

The motivation

Say you have a service registry in a dependency injection system. Components receive &ServiceRegistry — a shared reference — and ask it to resolve services by type. The registry lazily constructs each service the first time it's requested, then caches it in a HashMap. From the caller's perspective, resolve() is a read operation. But internally, the registry must insert into its cache. 

use std::collections::HashMap;
use std::any::{Any, TypeId};

struct ServiceRegistry {
    cache: HashMap<TypeId, Box<dyn Any>>,
}

impl ServiceRegistry {
    fn resolve<T: Any + Default>(&self) -> &T {
        let id = TypeId::of::<T>();
        if !self.cache.contains_key(&id) {
            self.cache.insert(id, Box::new(T::default())); // ERROR
        }
        self.cache.get(&id).unwrap().downcast_ref().unwrap()
    }
}

The compiler rejects this: 

error[E0596]: cannot borrow `self.cache` as mutable,
              as it is behind a `&` reference

The method takes &self, which is a shared, read-only reference. You can't write through it. That's not a suggestion — the borrow system depends on that guarantee.

Why the compiler does this

Rust's safety comes from two rules: if you have a &T, nobody is changing T right now. If you have a &mut T, nobody else can see T right now. These two rules prevent data races, iterator invalidation, and use-after-free — all at compile time, with no runtime cost.

The problem is that the HashMap really does need to change, even though from the outside get() looks like a read. Caching is an implementation detail the caller shouldn't care about. But the type system doesn't know that — it sees a write through a shared reference and says no.

In C++ terminology, this is the distinction between logical constness and physical constness. From the caller's perspective, the struct is logically const — calling get() doesn't change its observable behavior. But physically, the cache must be mutable. Rust's type system conflates the two by default; RefCell is the mechanism for separating them.

The pattern: RefCell<T>

RefCell<T> moves borrow checking from compile time to runtime. You wrap a value in it, then call .borrow() to read and .borrow_mut() to write. The same rules apply — many readers or one writer, never both — but they're checked when the code runs instead of when it compiles. 

use std::cell::RefCell;
use std::collections::HashMap;
use std::any::{Any, TypeId};

struct ServiceRegistry {
    cache: RefCell<HashMap<TypeId, Box<dyn Any>>>,
}

impl ServiceRegistry {
    fn resolve<T: Any + Default>(&self) -> T
    where T: Clone
    {
        let id = TypeId::of::<T>();
        if !self.cache.borrow().contains_key(&id) {
            self.cache.borrow_mut().insert(id, Box::new(T::default()));
        }
        self.cache.borrow().get(&id).unwrap().downcast_ref::<T>().unwrap().clone()
    }
}

The method still takes &self. The HashMap is wrapped in a RefCell, so writes go through it even behind a shared reference. The compiler accepts this because RefCell promises to enforce the borrow rules itself, at runtime. Multiple components receive &ServiceRegistry and call resolve() independently — the registry is logically const, but physically mutable.

What happens at runtime

Inside, RefCell keeps a counter. Each .borrow() increments it; dropping the returned guard decrements it. .borrow_mut() checks that the counter is zero — meaning nobody else is borrowing — and sets a flag. If you break the rules, like calling .borrow_mut() while a .borrow() guard is still alive, it panics. 

let cell = RefCell::new(42);

let a = cell.borrow();     // shared borrow — fine
let b = cell.borrow();     // second shared borrow — fine
let c = cell.borrow_mut(); // PANIC: already borrowed

That panic isn't a bug in RefCell (though it probably means one in your code). It's RefCell enforcing the same invariant the borrow checker verifies at compile time: you can't have a mutable borrow while shared borrows exist.

The tradeoff

With RefCell, you trade compile-time checking for runtime checking. Conflicting borrows are no longer rejected at compile time — you'll find out when your code panics. The borrow rules still apply; they just moved from the compiler to the running program.

The runtime cost is small — just checking an integer on each borrow. The real cost is the risk of panics. If you can take &mut self instead and let the compiler check everything statically, do that. Use RefCell when callers need shared access but the struct still needs to write to one of its fields.

What does this cost?

RefCell stores a single isize alongside your data — essentially a borrow counter. Each .borrow() checks and increments it; each guard drop decrements it. The check is a branch on an integer — nanoseconds. The runtime cost is negligible. The real cost is that a borrow violation becomes a panic instead of a compile error. Without RefCell, you'd need unsafe to achieve the same interior mutability — specifically, an UnsafeCell (which RefCell wraps internally) and manual tracking of the aliasing rules. RefCell is the safe abstraction over that unsafety.

When to use it

Good uses:

  • A service registry or resolver that caches results behind an &self method (the example above)
  • Fields that compute their value the first time they're accessed, then store it
  • A Vec of observers that gets modified even though the subject is shared
  • Any field that changes as an implementation detail, invisible to callers

When not to use it:

  • If you can change the method to take &mut self, do that — borrow violations are caught statically and there's no overhead
  • If you need thread safety, RefCell won't work — it's !Sync. Use Mutex or RwLock instead (#17)

What comes next

RefCell lets you write to a field through &self, but only one variable can own the struct. When multiple parts of your program need to hold onto the same data, you need Rc — which is the next post. And when you need multiple owners who can also write, #3 puts the two together.
