copied vs cloned

This iterator yields &u32. Make it yield u32 and signal the cost — Solution

March 2026 · Part B: The Solution

The answer: A

Use .copied().

copied() requires T: Copy. It converts an iterator of &T into an iterator of T by bitwise copying each element. For u32, that's 4 bytes — trivially cheap, no heap allocation, no deep clone. The method name signals this to the reader.

cloned() requires only T: Clone. When you see .cloned() in code, you can't tell at a glance whether the operation is cheap or expensive — Clone could mean a simple memcpy or a deep heap allocation. For u32, using cloned() sends a misleading signal: it suggests the copy might be costly when it isn't.

The solution

fn sum_even(numbers: &[u32]) -> u32 {
    numbers.iter()
        .filter(|&&n| n % 2 == 0)
        .copied()   // u32 is Copy — this is a trivial bitwise copy
        .sum()
}

fn main() {
    let data = vec![1, 2, 3, 4, 5, 6];
    println!("sum of evens: {}", sum_even(&data));  // 12
}

Why the other options fail

B. Keep cloned() — it compiles and produces the same result. But code communicates to humans, not just compilers. cloned() says "this clone might be expensive." copied() says "this is trivially cheap." When the compiler can enforce the cheap guarantee (via the Copy bound), use the method that communicates it.

C. map(|&n| n) — manual destructuring works but hides the intent. copied() is a named operation that says exactly what's happening: "dereference each reference and copy the value." Named adapters are more searchable, more readable, and more idiomatic than equivalent closures.

D. Remove it — filter yields references (&&u32). Without copied() or cloned(), sum() would need to work on references. While Sum is implemented for &u32, using copied() explicitly communicates the ownership transition.

If it compiles with copied(), prefer it over cloned(). It tells the reader: this copy is free.
