as_ / to_ / into_

This function takes &str but you have a String. Which conversion? — Solution

March 2026 · Part B: The Solution

The answer: A

Use message.as_str().

as_str() returns a &str view of the String's contents. Zero cost — no allocation, no copy. The as_ prefix is the naming convention for cheap borrows: a different view of the same data, nothing new created.

The original code calls .to_string() on a value that's already a String. That allocates a second String on the heap, copies the bytes, then immediately borrows the copy and drops it. Pure waste. When you just need to look at the data, use as_. When you need a new copy, use to_.

The solution

fn log_event(msg: &str) {
    println!("[EVENT] {}", msg);
}

fn main() {
    let message = format!("user {} logged in", 42);
    log_event(message.as_str());  // free borrow, no allocation
}

Note: &message also works via Deref coercion (&String → &str). Both are zero-cost. as_str() makes the intent explicit and demonstrates the as_ naming convention.

Why the other options fail

B. to_owned() — to_owned() creates a new owned copy. For a String, that means cloning the entire heap allocation. Then you'd borrow the clone. Same waste as the original bug, different method name.

C. into() — into() consumes the original value. String doesn't implement Into<&str> — you can't produce a borrow by consuming the owner, because the borrow would outlive the consumed value. The into_ prefix means "the original is gone." That's not what you want here.

D. to_str() — to_str() doesn't exist on String. And even if it did, the to_ prefix signals "create something new." For a free borrow, the convention is as_.

as_ = free peek. to_ = allocating copy. into_ = consuming transformation. Start with as_ and only upgrade when you need to.
