_or vs _or_else

Convert this Option to a Result. The error message needs computing — Solution

March 2026 · Part B: The Solution

The answer: A

Switch to ok_or_else with a closure.

ok_or converts an Option<T> to a Result<T, E> — Some(v) becomes Ok(v), None becomes Err(e). The problem is the same as unwrap_or: the error value is evaluated eagerly. That format! call allocates a String every time, even when the user exists and the error is never used.

ok_or_else defers the error construction to a closure. The format! only runs when the Option is None. On the happy path, zero allocation.

This is the same _or / _or_else rule applied to a different method. format! always allocates, so it always belongs behind _or_else.

The solution

use std::collections::HashMap;

fn find_user(users: &HashMap<u64, String>, id: u64) -> Result<&String, String> {
    users.get(&id)
        // format! only runs when the user doesn't exist
        .ok_or_else(|| format!("user {} not found", id))
}

fn main() {
    let mut users = HashMap::new();
    users.insert(1, "Alice".into());
    users.insert(2, "Bob".into());

    println!("{:?}", find_user(&users, 1));  // Ok, no error string allocated
    println!("{:?}", find_user(&users, 99)); // Err, format! runs here
}

Why the other options fail

B. Static string literal — changes the error message. You lose the user ID in the error, which makes debugging harder. The point is to keep the dynamic error message and avoid wasted allocation, not to simplify the error.

C. Use match — works but is verbose for a pattern that ok_or_else handles in one line. match is the escape hatch, not the first choice.

D. Always use ok_or_else — over-correction. For a static error like .ok_or("not found"), the eager version is cleaner and costs nothing. Use _or_else when the fallback does work; use _or when it doesn't.

ok_or for cheap errors, ok_or_else for computed ones — the same rule, applied to Option-to-Result conversion.
