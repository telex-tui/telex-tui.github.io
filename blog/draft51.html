A Lesson Series That Actually Explains Things

Part 1 — The Machine Before the Code

Before you write a single line, you need a mental map of what you're talking to. The STM32 is not a magic box that runs C. It's a Cortex-M core — an ARM design licensed to ST — attached to a bus matrix, and hanging off that bus matrix is a collection of peripherals: timers, GPIO ports, UARTs, ADCs, and so on. Each peripheral is a block of silicon with a set of registers mapped to a fixed address in the address space. The CPU doesn't "call" a peripheral — it reads and writes memory addresses, and the silicon responds.

The reference manual (RM) for your specific part is the document you will spend the most time in. Not the datasheet — that's pinout and electrical characteristics. The RM describes every register, every bit, every legal state transition. It's dense and dry and exactly right. Learning to navigate it is the actual skill this series is teaching. Everything else is application of that skill.

The memory map is your starting point. The RM will have a chapter showing you what lives at what address: FLASH at 0x08000000, SRAM at 0x20000000, and the peripheral registers scattered through 0x40000000 and up. The AHB and APB buses are the highways connecting the CPU to those peripherals, and they run at different speeds. When you write RCC->AHB1ENR |= ..., you are writing to a register at a specific address on a specific bus, and the silicon on the other end is changing its behaviour in response.

Get comfortable with the idea that the chip is a collection of state machines. Each peripheral has its own state, controlled by its registers. The CPU's job is to configure those state machines and then get out of the way — or to respond when they ask for attention.

Part 2 — Clocks Are Not Free

The STM32 does nothing without a clock. This sounds obvious but the implications run deep. Every peripheral on the chip is gated — it receives no clock signal until you explicitly enable it, which means it draws no power, but it also means its registers do nothing and reads return undefined values. Writing to a peripheral before enabling its clock is one of the most common silent failures for beginners.

The clock tree starts at an oscillator. You have choices: the internal HSI (typically 8 or 16 MHz, less accurate), the external HSE (whatever crystal you've fitted, accurate), or for USB and precise timing, a PLL. The PLL takes an input clock, divides it down to a reference frequency, multiplies it back up, and divides again to produce output clocks for the system and the buses. The RCC (Reset and Clock Control) peripheral is what you configure to set all of this up.

The RCC chapter of the RM will have a clock tree diagram — find it and print it. Every arrow on that diagram is a mux or divider that you control with RCC registers. The AHB prescaler, APB1 prescaler, APB2 prescaler — these determine the clock rates seen by each group of peripherals. Timers, in particular, have special rules about their clock source when the APB prescaler is not 1: the timer clock doubles. The RM documents this, but you have to read it.

When you set a baud rate for UART, or a PWM frequency for a timer, you are doing arithmetic against this clock tree. If you don't know what clock your peripheral is actually running from, your calculation is fiction. Always trace from the oscillator to the peripheral before you commit a frequency to code.

Part 3 — GPIO Is Just a Register

GPIO tutorials treat the LED blink as an incantation. It isn't. The GPIO peripheral has a small set of registers and once you understand them, everything else is obvious.

MODER — two bits per pin, setting the pin as input, output, alternate function, or analog. OTYPER — one bit per pin, push-pull or open-drain. OSPEEDR — two bits per pin, slew rate. PUPDR — two bits per pin, pull-up, pull-down, or none. ODR — one bit per pin, the output state. IDR — one bit per pin, the input state. BSRR — the atomic set/reset register: write a 1 to the low half to set a pin high, write a 1 to the high half to set it low. This is the register to use in interrupt contexts because the operation is atomic — no read-modify-write, no race.

The LED blink without HAL, for a pin on GPIOA bit 5, looks like this: enable GPIOA's clock in RCC, write the MODER to set pin 5 as output, write BSRR to toggle it. That's it. You are doing three register writes to real addresses and the LED lights up. There is no magic. When CubeMX generates ten function calls to achieve this, it's not wrong, it's just solving the wrong problem — code that configures everything generically and defensively at the cost of legibility.

Alternate function mode is worth understanding early. Every physical pin can be routed to a peripheral — a UART TX, a SPI clock, a timer output — depending on which alternate function you select in the AFRL or AFRH register. The datasheet's alternate function table tells you which AF number maps to which peripheral for each pin. When a peripheral "takes over" a pin, it's just the hardware mux switching. There is no magic there either.

Part 4 — Interrupts: The CPU's Trapdoor

The CPU is executing instructions sequentially. An interrupt is an asynchronous event that forces the CPU to abandon its current execution context, save enough state to resume later (the hardware does this automatically for the Cortex-M: it pushes a fixed set of registers onto the stack), and jump to a handler. When the handler returns, the CPU resumes where it left off.

The vector table is a table of function pointers sitting at the start of FLASH (or wherever the VTOR register points). Entry 0 is the initial stack pointer value. Entry 1 is the reset handler. Beyond that come the exception and interrupt vectors — one per source. When a particular interrupt fires, the hardware reads the corresponding entry from this table and jumps to it.

The NVIC (Nested Vectored Interrupt Controller) is the Cortex-M peripheral that manages interrupt enables and priorities. Before an interrupt source can fire, two things must be true: the peripheral must be configured to generate the interrupt (e.g., USART_CR1's RXNEIE bit), and the corresponding line in the NVIC must be enabled. Both gates must be open. Priority determines which interrupt wins when two fire simultaneously, and whether one interrupt can preempt another that's already running.

For GPIO interrupts, there's an extra layer: EXTI (External Interrupt/Event Controller). GPIO pins don't connect directly to NVIC lines. Instead, you configure which GPIO port's pin connects to an EXTI line (via SYSCFG), configure the edge trigger (rising, falling, both) in EXTI, and then enable the EXTI line in NVIC. When you understand this chain, the register configuration is mechanical.

The mistake people make in interrupt handlers is doing too much. A handler should record that an event occurred — set a flag, write to a ring buffer — and return. Heavy processing belongs in main-loop context. The moment you add blocking delays or complex logic to an interrupt handler, you've started building a disaster.

Part 5 — Timers Are Not Delays

The beginner's use of timers is HAL_Delay(), which spins on the SysTick counter and wastes CPU cycles. A timer is nothing like that. A timer is an independent counter peripheral that counts up (or down), compares its value against programmable thresholds, and generates events. The CPU can be asleep while this happens.

The core registers of a general-purpose timer are: PSC (prescaler — divides the input clock down before the counter sees it), ARR (auto-reload register — when the counter hits this value it resets to zero and optionally fires an interrupt), and CNT (the current count value). If your timer clock is 72 MHz, PSC=71 gives you a 1 MHz count rate, and ARR=999 gives you an event every millisecond. The timer does this on its own, indefinitely, without involving the CPU.

PWM is a natural consequence of this architecture, not a magic mode. Each timer has capture/compare channels. Each channel has a CCR register. When the counter value equals CCR, the output pin changes state. When the counter resets (ARR), it changes back. The duty cycle is CCR/ARR. Changing duty cycle means writing one register. The timer doesn't stop; the output just changes on the next cycle. This is why you can do smooth motor control or LED dimming with no CPU involvement after setup.

Input capture is the reverse: the timer records the value of CNT when a pin changes state. Period measurement and pulse width measurement fall out immediately. Output compare can trigger a DMA request rather than an interrupt, connecting the timer directly to the DMA engine without the CPU ever waking up. The timer is really a small state machine with several output modes — once you see it that way, the register names start to make sense.

Part 6 — UART From First Principles

UART is the simplest serial protocol and the right one to understand first because everything you learn here transfers to SPI and I2C.

The baud rate is the number of bits per second. To configure the STM32 UART, you write a value to BRR (Baud Rate Register). This value is derived from the peripheral clock divided by the desired baud rate. The RM gives you the formula; the key point is that if you have the wrong peripheral clock assumption, your baud rate will be wrong and your characters will be garbage. Trace the clock tree before you configure baud.

The status register (SR or ISR depending on the family) tells you what state the peripheral is in: TXE (transmit data register empty — safe to write another byte), RXNE (receive data register not empty — a byte has arrived), TC (transmission complete — the shift register is empty). The data register (DR or TDR/RDR) is what you read from and write to. Polling UART is a loop that checks TXE, writes a byte, repeats. It works. It also blocks the CPU for the duration of the transmission.

Interrupt-driven UART is the step up: enable RXNEIE in CR1, enable the UART interrupt in NVIC, and your handler fires whenever a byte arrives. You never poll. The transmit side is more subtle: you write the first byte to kick off transmission, then in the handler, write the next byte when TXE fires, and disable the TXE interrupt when you've sent the last one (otherwise it fires continuously because TXE is always true when the buffer is empty).

DMA-driven UART is the final form: you point a DMA channel at your buffer and the UART, specify the byte count, and the peripheral empties or fills the buffer with zero CPU involvement. The transfer-complete interrupt fires when it's done. This is how you do high-throughput serial without burning CPU cycles.

Part 7 — DMA: The CPU's Delegate

The CPU is one bus master on the AHB matrix. DMA is another. When DMA is active, it can move data between memory and peripheral registers — or between two memory locations — without involving the CPU at all. The CPU sets up the transfer and moves on.

A DMA channel (or stream, depending on the family) is configured with: the source address, the destination address, the transfer count, the direction (peripheral-to-memory or memory-to-peripheral), the data width (byte, halfword, word), and whether the address should auto-increment after each transfer. You enable the channel, and it runs. When the count reaches zero, a transfer-complete interrupt fires if you've enabled it.

The "request mapping" matters: each DMA channel can only serve certain peripherals. The RM has a table showing which request connects which peripheral to which DMA channel. In some STM32 families (F4 and later), you also select a "stream" and a "channel" within that stream. This is a source of legitimate confusion because the terms overload badly. Read the DMA chapter's request mapping table for your specific part.

Where DMA breaks: if the source and destination are in the same bus domain and both need the bus simultaneously, one will stall. If you're moving data into SRAM from a peripheral on APB1, and the CPU is simultaneously doing heavy work from SRAM, you'll see contention. The memory architecture chapter of the RM will tell you which buses can operate in parallel. Double-buffering (using DMA's circular mode or manually flipping between two buffers) is the standard answer for continuous streaming.

Cache coherency becomes a problem on higher-end parts (F7, H7) that have data caches. If DMA writes to memory that's cached from the CPU's perspective, the CPU may read stale data. You have to explicitly invalidate or clean cache lines around DMA transfers. This is a real source of hard-to-diagnose bugs.

Part 8 — SPI and I2C: Protocol Before Peripheral

The mistake is to start with the peripheral. Start with the electrical contract.

SPI has four wires: SCLK (clock, driven by master), MOSI (master out slave in), MISO (master in slave out), and CS (chip select, active low, driven by master). Data is exchanged simultaneously in both directions on every clock edge — one bit out, one bit in. The clock polarity (CPOL) and phase (CPHA) determine whether data is sampled on the rising or falling edge. These two bits define the four SPI modes (0–3); your device's datasheet will tell you which mode it needs. Get this wrong and every byte is garbage.

The STM32 SPI peripheral is a shift register. You write a byte to DR, it clocks out while simultaneously clocking in, and when done, the received byte is in DR. In polling mode you wait for TXE to write and RXNE to read. The peripheral handles the clock and the bit-banging. CS is typically controlled manually (write a GPIO low, do the transfer, write it high), though the NSS pin can be hardware-controlled with caveats.

I2C is more complex. It's a two-wire bus (SDA, SCL) with open-drain outputs — multiple devices share the bus, any device can pull a line low, nobody drives it high directly (the pull-up resistor does that). Every transaction begins with a START condition (SDA falls while SCL is high), then an address byte (7 or 10 bit) with a direction bit, then data bytes each acknowledged by the receiver, then a STOP. Address collisions and clock stretching are part of the protocol.

The STM32 I2C peripheral on F1/L1 families is notoriously awkward to drive correctly — the state machine has timing-sensitive sequences that are easy to miss. The F3/F4 and later families use a revised peripheral (I2C v2) with a much cleaner NBYTES/AUTOEND model that makes the driver significantly less fragile. Know which one you have. In both cases, the principle holds: if you understand what the wire is supposed to look like, you can read the register documentation with a clear goal in mind and verify your implementation with a logic analyser.

Part 9 — The HAL Is a Map, Not the Territory

HAL (Hardware Abstraction Layer) exists for a real reason: it lets ST write middleware (USB stacks, FAT filesystems, protocol implementations) that runs on any supported part without changes. It also makes CubeMX-generated initialisation code portable. These are legitimate goals.

The problem is that HAL is often presented as the way to program STM32, obscuring what's actually happening. HAL_GPIO_WritePin() is three or four register writes and a function call overhead. That overhead is irrelevant in most contexts, but it's also irrelevant in the other direction — once you know what the function does, there's nothing mystical about it. The function is a map to the territory. The territory is the register.

Reading CubeMX-generated code is a useful exercise not to copy it, but to decode it. When you see __HAL_RCC_GPIOA_CLK_ENABLE(), expand that macro. It's setting a bit in RCC->AHB1ENR. When you see HAL_UART_Init(), trace it into the HAL source (ST ships it) and find the register writes. You are building a translation table between HAL names and RM concepts. After doing this a few times, HAL code becomes legible rather than opaque.

Where HAL genuinely saves time: USB, Ethernet, and complex protocol drivers where the state machine is large and correctness matters more than the last few cycles. Where it costs you: timing-critical paths, interrupt latency analysis, power optimisation, and any situation where you need to know exactly what the silicon is doing and when. The engineers who use HAL effectively are the ones who don't need it — they could write the registers directly, they just choose not to for certain tasks.

LL (Low-Layer) drivers are the middle ground: thin inline functions that map almost directly to registers, with ST's naming and bit definitions, but without the state machine overhead of HAL. Worth knowing they exist.

Part 10 — Startup, Linker Scripts, and the World Before main()

Your C program doesn't start at main(). It starts at the reset handler — a function specified in the vector table that runs before anything else. What it does:

  1. Sets the stack pointer (the hardware loads this from the vector table's first entry, but the startup code may verify it).
  2. Copies the .data section from FLASH to SRAM. Initialised global variables live in FLASH as their initial values; at startup they must be copied to SRAM where the CPU can write them.
  3. Zeroes the .bss section. Uninitialised globals are required by C to be zero; the startup code writes zeros across that memory range.
  4. Calls static constructors if using C++ (__libc_init_array).
  5. Calls main().

The linker script is what defines where .data, .bss, .text, and the vector table live in the address space. It specifies FLASH and SRAM regions and assigns output sections to them. It also defines the symbols that the startup code uses to know where .data starts in FLASH and where it should be copied to in SRAM. If you've ever wondered why a global variable initialised to zero works differently from one explicitly initialised to 0 in terms of binary size — the linker script is why.

Understanding this matters when you use sections other than the defaults: putting a function in SRAM for execution-from-RAM (because SRAM is faster or because you need to reprogram FLASH while running), using CCM (Core Coupled Memory) on F4 parts for latency-critical buffers, or setting up a bootloader that must leave a defined region of SRAM intact across a reset. The linker script is a programming language for memory layout, not a configuration file you import blindly.

Part 11 — Power, Sleep, and the Real-Time Budget

Battery-powered embedded systems live and die on sleep modes. The STM32 has several: Sleep (CPU stopped, peripherals running), Stop (most clocks stopped, SRAM retained, some peripherals like RTC and EXTI can still wake), and Standby (almost everything off, only the backup domain alive). Each mode trades current draw against wake latency and which wakeup sources are available.

The SysTick timer is the Cortex-M's built-in periodic interrupt — conventionally ticking every 1ms to drive HAL's timebase. In Sleep mode, SysTick still runs and wakes the CPU every millisecond. If your application spends most of its time waiting for an external event, this is wasteful. You either stop SysTick and use an RTC or LPTIM as the wakeup source, or you run tickless idle (FreeRTOS supports this) where the OS programs the timer for the next scheduled wakeup and enters sleep until then.

Power budgeting is arithmetic. Your peripheral draws X μA. Your MCU in Stop mode draws Y μA. Your wake interval is Z milliseconds and each wake takes W milliseconds to do useful work. Average current is (X+MCU_active)*W/Z + Y*(1-W/Z). If Z is 1 second and W is 1 millisecond, you're asleep 99.9% of the time. The ratio of active to sleep current determines whether you can run for months on a coin cell or need a battery pack.

Clock selection interacts with power. Starting from Stop mode with HSI is fast (a few microseconds) but HSI is less accurate and lower frequency than your full PLL. Starting from HSE and locking the PLL takes longer and costs current during the lock time. Knowing your application's timing tolerance determines which tradeoff is correct. This is engineering, not configuration.

Part 12 — Debugging Without Guessing

The STM32 connects to a debugger via SWD (Serial Wire Debug) — a two-wire protocol (SWDIO + SWCLK) plus reset. OpenOCD is the open-source host-side interface; GDB connects to OpenOCD and gives you breakpoints, watchpoints, memory inspection, and register reads. A cheap ST-LINK V2 clone is sufficient for most work. If you're using a Nucleo board, there's an onboard ST-LINK that enumerates as a debugger over USB.

The first thing to learn in GDB is info registers and how to read PC (program counter), SP (stack pointer), and LR (link register) when something has crashed. The second is x/32xw 0x20000000 to dump memory. The third is setting hardware breakpoints (hbreak) which work in FLASH without modifying the code, unlike software breakpoints which patch instructions.

When the CPU takes a fault — HardFault, BusFault, MemManageFault, UsageFault — it doesn't just stop. It jumps to the corresponding handler. The default handler in most startup files is an infinite loop, which is useless. Write your own. In a real fault handler, you have access to the stacked register frame (the CPU pushed it before jumping), which gives you the PC at the time of the fault, the LR, and the offending address for memory faults. The SCB (System Control Block) has fault status registers: CFSR, HFSR, MMAR, BFAR. These tell you whether the fault was an undefined instruction, an unaligned access, a bus error on a specific address, or something else. Reading these registers from the fault handler turns a hard crash into a specific diagnosis.

A fault handler that prints to UART or toggles an LED in a pattern is a crutch. A fault handler that records the registers and calls a breakpoint instruction (BKPT 0) if a debugger is attached, or resets and logs to FLASH if not, is a tool. The difference between guessing and debugging is whether you have data.

Coda: The Reference Manual Is the Source of Truth

Every one of these lessons points at the same document: the RM for your part. ST produces one for each family — RM0008 for F1, RM0090 for F4, and so on. They are hundreds or thousands of pages. You don't read them cover to cover. You read the chapter for the peripheral you're working on, follow its cross-references to the clock chapter, and look up the register definitions when you're writing code.

The habit to build is: when something doesn't work, go to the RM before you go to Stack Overflow. Stack Overflow will tell you what code to paste. The RM will tell you why it works, which is the only knowledge that transfers to the next problem.
