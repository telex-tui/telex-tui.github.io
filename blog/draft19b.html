You need a graph. Some nodes have two parents — Solution

March 2026 · Part B: The Solution

The answer: A

Store all packages in a Vec and use indices.

Graphs have shared ownership by definition — a node can be reachable from multiple other nodes. Rust's ownership model is a tree: one owner, no sharing. These two models conflict. The arena-and-indices pattern resolves the conflict by making the Vec the single owner of all nodes, and using plain integers as edges.

An index is just a number. It doesn't borrow anything. It doesn't create lifetime constraints. Two nodes can both store the index 3 to point to the same package, and there's no ownership conflict because neither of them owns the package — the arena does.

This is the standard pattern for graphs in Rust. The petgraph crate uses it. Compilers use it for ASTs. ECS game engines use it for entity relationships. It works because it separates storage (the arena owns everything) from structure (indices describe relationships).

The solution

use std::collections::HashMap;

type PackageId = usize;

struct Package {
    name: String,
    deps: Vec<PackageId>,
}

struct Graph {
    packages: Vec<Package>,
    by_name: HashMap<String, PackageId>,
}

impl Graph {
    fn new() -> Self {
        Graph { packages: vec![], by_name: HashMap::new() }
    }

    fn add(&mut self, name: &str) -> PackageId {
        if let Some(&id) = self.by_name.get(name) {
            return id;
        }
        let id = self.packages.len();
        self.packages.push(Package { name: name.into(), deps: vec![] });
        self.by_name.insert(name.into(), id);
        id
    }

    fn add_dep(&mut self, from: PackageId, to: PackageId) {
        self.packages[from].deps.push(to);
    }
}

fn main() {
    let mut g = Graph::new();
    let serde = g.add("serde");
    let web = g.add("web-framework");
    let app = g.add("my-app");

    // Both app and web-framework depend on serde
    g.add_dep(app, serde);
    g.add_dep(web, serde);
    g.add_dep(app, web);

    for pkg in &g.packages {
        println!("{} depends on {:?}", pkg.name, pkg.deps);
    }
}

Why the other options fail

B. Rc<RefCell<>> everywhere — works for DAGs, but if your graph has cycles (A depends on B, B depends on A), neither node's reference count reaches zero. Memory leak. Also adds runtime overhead for borrow checking you don't need if you're just reading the graph.

C. Raw pointers — requires unsafe for every access. You become responsible for ensuring pointers aren't dangling. One mistake and you have use-after-free. The arena pattern gives you the same flexibility with zero unsafe.

D. References with lifetimes — you can't express "node A points to node B, and node B points to node A" with lifetimes. Each reference needs to outlive what it points to, creating a circular dependency that lifetimes can't represent.

Graphs in Rust: store nodes in a Vec, use indices as edges.
