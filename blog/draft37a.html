_or vs _or_else

This fallback allocates a String every call. It shouldn't

March 2026 · Part A: The Challenge

A function that reads a config value with a fallback.

use std::collections::HashMap;

fn get_config_value(config: &HashMap<String, String>, key: &str) -> String {
    config.get(key)
        .cloned()
        .unwrap_or(String::from("default_value"))
}

fn main() {
    let mut config = HashMap::new();
    config.insert("host".into(), "localhost".into());
    config.insert("port".into(), "8080".into());

    // This key exists — but the fallback String is still allocated
    println!("{}", get_config_value(&config, "host"));
    println!("{}", get_config_value(&config, "missing"));
}

The String::from("default_value") runs on every call, even when the key exists. Fix the unnecessary allocation.

Your options

  • Switch to unwrap_or_else with a closure that builds the String only when needed
  • Use match instead — pattern matching avoids this problem entirely
  • Keep unwrap_or but pass a string literal &str instead of a String
  • Use unwrap_or_default() since it always avoids allocation

Pick one before reading the solution.
