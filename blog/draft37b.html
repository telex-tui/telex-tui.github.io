_or vs _or_else

This fallback allocates a String every call. It shouldn't — Solution

March 2026 · Part B: The Solution

The answer: A

Switch to unwrap_or_else.

unwrap_or evaluates its argument eagerly. That String::from("default_value") runs before Rust checks whether the Option is Some or None. When the key exists, the freshly allocated String is immediately dropped. Pure waste.

unwrap_or_else takes a closure. The closure only runs when the Option is None. When the key exists, the closure never executes and the String is never allocated. Same result, zero wasted work on the happy path.

This is the core rule of the _or / _or_else pair: if the fallback is cheap (a literal, a Copy value, something already computed), use _or. If the fallback allocates, calls a function, or does any real work, use _or_else.

The solution

use std::collections::HashMap;

fn get_config_value(config: &HashMap<String, String>, key: &str) -> String {
    config.get(key)
        .cloned()
        // Closure only runs when the key is missing
        .unwrap_or_else(|| String::from("default_value"))
}

fn main() {
    let mut config = HashMap::new();
    config.insert("host".into(), "localhost".into());
    config.insert("port".into(), "8080".into());

    println!("{}", get_config_value(&config, "host"));    // no wasted allocation
    println!("{}", get_config_value(&config, "missing")); // allocates here, as needed
}

Why the other options fail

B. Use match instead — match works and avoids eager evaluation, but it's more verbose for the same outcome. unwrap_or_else exists precisely to express this pattern in one line. Reaching for match when a combinator fits is idiomatic regression.

C. Pass a &str literal — the function returns String. You can't pass a &str where a String is expected. And even if you restructured the return type, you'd be changing the API to avoid learning the idiom.

D. unwrap_or_default() — String::default() returns an empty string, not "default_value". It avoids the allocation issue, but it gives the wrong answer. And it still allocates an empty String on the heap (though that's typically optimised away).

If the fallback does work, wrap it in _or_else so the work only happens when needed.
