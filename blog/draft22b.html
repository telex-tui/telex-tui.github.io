This value needs to live longer than the function that created it — Solution

March 2026 · Part B: The Solution

The answer: A

Use a move closure to transfer ownership into the thread.

The function creates the config. The thread needs the config. The function returns before the thread finishes. So the config can't live on the function's stack — it needs to move somewhere that outlives the function. The move keyword on the closure transfers ownership of config from the function's stack into the closure, which is then owned by the thread.

After the move, the closure owns the config. The thread owns the closure. So the config lives as long as the thread does. No reference counting, no leaking, no unsafe. Just ownership transfer.

This is the simplest case of "extending a value's lifetime" — you don't extend it at all. You move it to where it needs to be. The value doesn't live longer; it lives somewhere else.

The solution

use std::thread;

struct DatabaseConfig {
    host: String,
    port: u16,
}

fn start_worker() -> thread::JoinHandle<()> {
    let config = DatabaseConfig {
        host: "localhost".into(),
        port: 5432,
    };

    // move transfers ownership of config into the thread
    thread::spawn(move || {
        println!("Worker using {}:{}", config.host, config.port);
    })
}

fn main() {
    let handle = start_worker();
    handle.join().unwrap();
}

Why the other options fail

B. Annotate with 'static — 'static is a lifetime bound, not a lifetime extension. You can't make a stack-allocated value live forever by annotating it. The 'static bound means "this value contains no non-static references" — it doesn't change where the value is stored or how long it lives.

C. Box::leak — makes the config live permanently by leaking the heap memory. It never gets freed. This works for truly global singletons but is wildly inappropriate for a per-worker config. Start 1000 workers and you've leaked 1000 configs.

D. Arc — shared ownership between the function and the thread. But the function doesn't need the config after spawning the thread. Arc is for when multiple owners need simultaneous access. When there's only one consumer, just move.

When only one consumer needs the data, move ownership to it — don't share, don't leak, just move.
