You clone everywhere to make it compile. What's the real fix? — Solution

March 2026 · Part B: The Solution

The answer: A

Scope the borrow. Extract what you need, then mutate.

The problem is that items.iter() borrows the vec immutably, and items.push() borrows it mutably. The borrow checker won't let both coexist. But they don't need to coexist — you finish reading before you start writing.

The fix is to extract the data you need from the immutable borrow into a local variable that doesn't reference the vec at all. Once you have the length (a usize, which is Copy) and the formatted string, the borrow on items is released, and you're free to push.

This is the "split the read phase from the write phase" pattern. Almost every "borrow checker fight" involving collections comes down to this: you're reading and writing the same thing in overlapping scopes. Separate them and the conflict disappears.

The solution

fn main() {
    let mut items = vec![
        String::from("apple"),
        String::from("banana"),
        String::from("cherry"),
    ];

    // Read phase: find the longest, build the new string
    // The borrow of `items` ends when this statement completes
    let new_entry = {
        let longest = items.iter()
            .max_by_key(|s| s.len())
            .unwrap();
        format!("best: {}", longest)
    };

    // Write phase: borrow is gone, safe to mutate
    items.push(new_entry);

    for item in &items {
        println!("{}", item);
    }
}

Why the other options fail

B. Rc<String> — adds heap allocation and reference counting overhead to every string in the collection, just to share one reference. The underlying problem isn't ownership — it's borrow scope. Using Rc here is like buying a crane to move a chair.

C. Cow<str> — Cow helps when you sometimes borrow and sometimes own. That's not the problem here. The problem is overlapping borrows, and Cow doesn't solve that.

D. Accept the clone — cloning works, but it's a workaround, not a solution. The clone allocates a new heap string when you don't need to. Learning to scope borrows is the skill; reaching for .clone() is the crutch.

When the borrow checker blocks you, separate the read phase from the write phase.
