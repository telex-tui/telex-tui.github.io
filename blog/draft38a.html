_or vs _or_else

Convert this Option to a Result. The error message needs computing

March 2026 · Part A: The Challenge

A lookup function that returns a Result with a descriptive error.

use std::collections::HashMap;

fn find_user(users: &HashMap<u64, String>, id: u64) -> Result<&String, String> {
    users.get(&id)
        .ok_or(format!("user {} not found", id))
}

fn main() {
    let mut users = HashMap::new();
    users.insert(1, "Alice".into());
    users.insert(2, "Bob".into());

    println!("{:?}", find_user(&users, 1));
    println!("{:?}", find_user(&users, 99));
}

The format! macro allocates a String on every call, even when the user exists. Fix it.

Your options

  • Switch to ok_or_else with a closure that calls format! only on None
  • Use ok_or with a static string literal instead of format!
  • Use match to handle Some and None separately
  • Use ok_or_else even for static error messages to be safe

Pick one before reading the solution.
