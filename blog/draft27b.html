This struct holds a reference. Now give it a method — Solution

March 2026 · Part B: The Solution

The answer: A

Write fn next_word(&mut self) -> Option<&str> and let lifetime elision handle it.

Here's the trick: the returned &str points into self.input, which has lifetime 'a. But you don't need to spell that out. Rust's lifetime elision rule for methods says: if there's a &self or &mut self, the output lifetime defaults to the lifetime of self. Since the returned slice comes from data inside self, this default is correct.

The method takes &mut self because it advances self.pos. But the returned &str borrows from the input, not from the parser's mutable state. The borrow checker sees that the returned slice points into the immutable input field and is fine with it.

Don't overthink lifetimes on methods. Write the obvious signature first. If the compiler complains, add annotations. Most of the time, elision gets it right.

The solution

struct Parser<'a> {
    input: &'a str,
    pos: usize,
}

impl<'a> Parser<'a> {
    fn new(input: &'a str) -> Self {
        Parser { input, pos: 0 }
    }

    // Elision assigns the return lifetime to &self's lifetime,
    // which is connected to 'a through the struct
    fn next_word(&mut self) -> Option<&str> {
        let remaining = &self.input[self.pos..];
        let trimmed = remaining.trim_start();
        if trimmed.is_empty() {
            return None;
        }
        let start = self.input.len() - trimmed.len();
        let end = trimmed.find(' ').map(|i| start + i).unwrap_or(self.input.len());
        self.pos = end;
        Some(&self.input[start..end])
    }
}

fn main() {
    let source = String::from("hello world rust");
    let mut parser = Parser::new(&source);

    while let Some(word) = parser.next_word() {
        println!("word: {}", word);
    }
}

Why the other options fail

B. fn next_word(&'a mut self) — this ties the mutable borrow of self to the lifetime 'a (the input's lifetime). That means you can't call next_word again while a previous result is alive — the mutable borrow of the parser lasts as long as the input. You'd only be able to call the method once.

C. Clone the input — throws away the zero-copy parsing advantage. If the input is a 10MB string, you've just allocated 10MB to avoid writing a lifetime.

D. Second lifetime parameter — unnecessary complexity. Two lifetimes are needed when the method's borrow and the data's borrow are genuinely independent. Here they're not — elision handles the relationship correctly.

Trust lifetime elision on methods — write the obvious signature first, annotate only when the compiler asks.
