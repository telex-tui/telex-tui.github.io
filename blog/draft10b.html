Two functions need the same string — Solution

March 2026 · Part B: The Solution

The answer: B

Change both functions to accept &str and pass references.

Neither function needs to own the string. They read it, print it, and move on. When a function only needs to read data, it should borrow it. That's the whole point of references — they let you share access without transferring ownership.

The borrow checker sees this clearly: greeting lives in main, and both function calls borrow it for the duration of their execution. The borrows don't overlap. No conflict.

This is the default instinct you want to build. When a function just reads a value, take &str instead of String, take &[T] instead of Vec<T>, take &T instead of T. Only take ownership when you need to store the value or move it somewhere else.

The solution

fn main() {
    let name = String::from("Alice");
    let greeting = format!("Hello, {}!", name);

    // Both functions borrow greeting — no move, no clone
    log_message(&greeting);
    send_to_client(&greeting);
}

fn log_message(msg: &str) {
    println!("[LOG] {}", msg);
}

fn send_to_client(msg: &str) {
    println!("[SEND] {}", msg);
}

Why the other options fail

A. Clone the string — works, but allocates a second copy of the data on the heap for no reason. Cloning is a code smell when borrowing solves the problem.

C. Wrap in Rc — reference counting is for shared ownership across complex lifetimes. Two sequential function calls don't need shared ownership. This is using a sledgehammer to hang a picture frame.

D. Return the string back — technically works but makes the API awkward and fragile. Every consumer becomes part of an ownership relay chain. It doesn't scale past two functions.

If you don't need to own it, borrow it.
