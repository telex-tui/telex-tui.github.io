as_ / to_ / into_

You need the raw bytes of a String. Borrow or consume? — Solution

March 2026 · Part B: The Solution

The answer: A

Use payload.as_bytes().

as_bytes() returns &[u8] — a borrowed view of the String's internal buffer. Zero cost. No allocation, no copy, no ownership change. The as_ prefix tells you: this is just a peek at the same data through a different lens.

The original code clones the String (heap allocation), then consumes the clone with into_bytes() to get a Vec<u8>. But checksum takes &[u8] — a borrowed slice. There was never any need for owned bytes. When a function only needs to read, borrow with as_.

The solution

fn checksum(data: &[u8]) -> u32 {
    data.iter().map(|&b| b as u32).sum()
}

fn main() {
    let payload = String::from("hello world");
    println!("checksum: {}", checksum(payload.as_bytes()));  // zero-cost borrow
    println!("original: {}", payload);                       // still available
}

Why the other options fail

B. Keep the clone — consuming is not the only way to get bytes. as_bytes() borrows them. The clone exists because into_bytes() consumes the String, but that's the wrong tool when you only need to read. Choose as_ (borrow) over into_ (consume) when you still need the original.

C. as_bytes().to_vec() — as_bytes() borrows for free, then to_vec() copies all the bytes into a new Vec. checksum takes &[u8], so owned bytes aren't needed. The to_ prefix warns you: this allocates. Don't upgrade from borrow to owned unless the API demands it.

D. into_bytes() without clone — into_bytes() consumes the String. The second println! uses payload — it won't compile because payload has been moved. The into_ prefix means: the original is gone.

as_ borrows for free. to_ copies and allocates. into_ consumes the original. Always start with as_ and only escalate when ownership demands it.
