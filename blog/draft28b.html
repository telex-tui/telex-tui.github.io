Two structs reference each other. Model that without cycles — Solution

March 2026 · Part B: The Solution

The answer: A

Store connections in a HashMap on the Server. Give each connection a server name (or ID) rather than a pointer back.

The bidirectional reference is an ownership cycle: Server owns Connections, Connections point to Server. In Rust, you break cycles by removing one direction of ownership. The server owns the connections (natural — it manages them). The connections don't own or borrow the server — they just know its name.

If a connection needs to do something with the server, the code that has access to both passes the server reference in. This is the "pass context at the call site" pattern rather than "store context in every struct." It's more explicit, but it avoids all the lifetime and cycle problems.

This is the same idea as the arena pattern, but applied to a two-type relationship. Ownership flows one direction; cross-references are IDs or keys, not pointers.

The solution

use std::collections::HashMap;

type ConnectionId = u64;

struct Server {
    name: String,
    max_connections: usize,
    connections: HashMap<ConnectionId, Connection>,
}

struct Connection {
    id: ConnectionId,
    remote_addr: String,
    server_name: String, // just the name, not a reference
}

impl Server {
    fn new(name: &str, max: usize) -> Self {
        Server {
            name: name.into(),
            max_connections: max,
            connections: HashMap::new(),
        }
    }

    fn accept(&mut self, id: ConnectionId, addr: &str) {
        let conn = Connection {
            id,
            remote_addr: addr.into(),
            server_name: self.name.clone(),
        };
        self.connections.insert(id, conn);
    }
}

fn main() {
    let mut server = Server::new("web-01", 100);
    server.accept(1, "192.168.1.5:4321");
    server.accept(2, "10.0.0.3:9999");

    for (id, conn) in &server.connections {
        println!("conn {} on {} from {}", id, conn.server_name, conn.remote_addr);
    }
}

Why the other options fail

B. Rc in both directions — Server holds Rc<Connection>, Connection holds Rc<Server>. Neither reference count reaches zero. Memory leak. You'd need Weak for one direction, adding complexity and runtime checks.

C. Raw pointers — requires unsafe everywhere. One mismanaged pointer and you have use-after-free or dangling references. No reason to accept this risk when ID-based lookup is simple and safe.

D. &Server with lifetime — the Connection can't outlive the borrow of Server, and you can't mutate the Server while any Connection borrows it. Adding or removing connections becomes impossible while any connection holds a reference to the server.

Break bidirectional ownership by storing IDs instead of pointers in one direction.
