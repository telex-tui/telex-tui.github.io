Add logging to this without changing every call site — Solution

March 2026 · Part B: The Solution

The answer: B

Put the logger inside the Client struct.

Every function already takes &Client. If the logger lives inside the client, every call site automatically gets logging without changing its signature. The logging happens inside get() — the method that actually makes the request. The caller doesn't need to know or care about logging.

This is the "thread context through a struct" pattern. Instead of passing five arguments to every function (client, logger, metrics, config, tracer), you bundle them into the struct that's already being passed around. In Rust, this is especially natural because the struct already has a clear ownership story — whoever owns the Client also owns the logger.

No new lifetimes, no new trait bounds, no global state. Just a field on a struct.

The solution

struct Logger {
    prefix: String,
}

impl Logger {
    fn new(prefix: &str) -> Self {
        Logger { prefix: prefix.into() }
    }

    fn log(&self, msg: &str) {
        println!("[{}] {}", self.prefix, msg);
    }
}

struct Client {
    base_url: String,
    logger: Logger, // logger lives inside the client
}

impl Client {
    fn new(base_url: &str) -> Self {
        Client {
            base_url: base_url.into(),
            logger: Logger::new("HTTP"),
        }
    }

    fn get(&self, path: &str) -> String {
        let url = format!("{}{}", self.base_url, path);
        self.logger.log(&format!("GET {}", url));
        url
    }
}

fn fetch_users(client: &Client) -> String {
    client.get("/users")
}

fn fetch_orders(client: &Client) -> String {
    client.get("/orders")
}

fn main() {
    let client = Client::new("https://api.example.com");
    println!("{}", fetch_users(&client));
    println!("{}", fetch_orders(&client));
}

Why the other options fail

A. Add &Logger parameter everywhere — this is exactly what the challenge says not to do. Every function that uses the client would need a new parameter. It doesn't scale.

C. Global static mut — every access requires unsafe, and concurrent access is undefined behaviour. Even in single-threaded code, static mut is a red flag. If you need a global logger, use the log crate.

D. Logging trait with default method — over-engineered. You'd need to define a trait, implement it, and call the trait method instead of a regular method. It adds indirection without solving any real problem. A field on the struct does the job.

When every call site already passes a struct, put new cross-cutting concerns inside that struct.
