_by / _by_key

Find the process using the most memory — Solution

March 2026 · Part B: The Solution

The answer: A

Use max_by_key.

max_by_key extracts a comparable value from each element and returns the element with the maximum key. One pass, no mutation, no manual tracking. The _by_key suffix means "compare by this extracted property" — you point at the field and the adapter handles the comparison.

This is the same _by / _by_key pattern from sorting. sort_by_key, min_by_key, max_by_key — all work the same way. Extract a key, compare by it.

The solution

struct Process {
    name: String,
    memory_mb: u64,
}

fn main() {
    let procs = vec![
        Process { name: "nginx".into(), memory_mb: 120 },
        Process { name: "postgres".into(), memory_mb: 450 },
        Process { name: "redis".into(), memory_mb: 85 },
    ];

    let heaviest = procs.iter()
        .max_by_key(|p| p.memory_mb)
        .unwrap();

    println!("{}: {} MB", heaviest.name, heaviest.memory_mb);
    // postgres: 450 MB
}

Why the other options fail

B. max_by with a full comparator — max_by(|a, b| a.memory_mb.cmp(&b.memory_mb)) works but is verbose for a single field. You're writing the comparison logic that _by_key handles for you. Use max_by when the comparison involves multiple fields or custom logic. For one field, _by_key is cleaner.

C. max() — max() requires Process: Ord. Custom types don't implement Ord by default, and deriving it would compare all fields in declaration order — not what you want. _by_key lets you pick exactly which property to compare without implementing traits.

D. sort_by_key then take last — sorting is O(n log n). Finding the maximum is O(n). Don't sort when you only need the extremum. max_by_key makes a single linear pass — the right tool for "find the biggest."

_by_key extracts and compares in one step. Use it for sort, min, max — anywhere you need "by this field."
