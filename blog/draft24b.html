Walk this tree without cloning the nodes — Solution

March 2026 · Part B: The Solution

The answer: A

Take &FileNode and iterate with .iter().

The original function takes FileNode by value — it consumes the node. That's why the caller has to clone: to keep a copy for itself. The fix is to take a reference. The function only needs to read the nodes, so a shared & reference is enough.

When you call .iter() on Vec<Box<FileNode>>, you get &Box<FileNode> for each element. Rust auto-derefs that to &FileNode when you pass it to the recursive call. No explicit dereferencing needed.

This is one of the most common Rust corrections: you wrote a function that takes ownership when it only needed a reference. The compiler told you with a move error. The fix is almost always to add &.

The solution

struct FileNode {
    name: String,
    children: Vec<Box<FileNode>>,
}

// Takes &FileNode — borrows, doesn't consume
fn print_tree(node: &FileNode, depth: usize) {
    println!("{}{}", "  ".repeat(depth), node.name);
    for child in &node.children {
        print_tree(child, depth + 1);
    }
}

fn main() {
    let tree = FileNode {
        name: "src".into(),
        children: vec![
            Box::new(FileNode { name: "main.rs".into(), children: vec![] }),
            Box::new(FileNode { name: "lib.rs".into(), children: vec![] }),
        ],
    };

    print_tree(&tree, 0);
    // tree is still fully owned here — no clone needed
    println!("tree still has {} children", tree.children.len());
}

Why the other options fail

B. Flatten into a Vec with indices — restructures the entire data model to solve a problem that just needs &. The arena pattern is great for graphs, but overkill for a simple read-only traversal of a tree you already have.

C. Rc<FileNode> — adds reference counting overhead to every node. Shared ownership is for when multiple parts of the program need to own the same data simultaneously. A read-only walk doesn't need shared ownership — it needs a reference.

D. Explicit stack with owned nodes — popping nodes off a stack moves them out of the tree, destroying the original structure. The caller loses the tree after traversal, which is the same problem as the original code.

If you're just reading, take a reference — don't clone, don't share, don't restructure.
