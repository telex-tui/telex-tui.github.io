How Polars Works Internally

March 2026

Draft — work in progress, not for publication.

Polars is a high-performance DataFrame library with a Python API and a Rust engine. A DataFrame, in Polars and elsewhere, is a table of named, typed columns; a Series is a single column. The bridge between the two languages is PyO3, a Rust crate that generates the FFI layer letting Python call directly into compiled Rust code.

When Python runs import polars, it loads a compiled shared library (.so on Linux, .dylib on macOS) containing the entire Rust engine. Python and Rust share the same process and address space from that point on. A "boundary crossing" is just a function call across the language boundary, with type conversion at each end — not IPC, not a network call. Python's GIL (Global Interpreter Lock) means only one thread can run Python bytecode at a time, but Rust code can release the GIL to do heavy computation without blocking Python threads. This turns out to be critical for performance, and we'll see exactly where it happens.

Everything below is based on reading the actual Polars source code as of early 2026, not documentation or blog posts. File paths are accurate at time of writing and subject to change — treat them as pointers, not precise references.

The simplest possible round trip — df.height

The number of rows in a DataFrame. It's the simplest property Polars exposes — a single field read — which makes it the cleanest trace through the full call stack, from Python property to Rust struct field.

Step 1: Python property — py-polars/src/polars/dataframe/frame.py:832–846 

@property
def height(self) -> int:
    return self._df.height()

self._df is typed as PyDataFrame (declared at line 372: _df: PyDataFrame). This is the Rust struct exposed to Python via PyO3.

Step 2: PyO3 boundary — crates/polars-python/src/dataframe/general.rs:183–185 

pub fn height(&self) -> usize {
    self.df.read().height()
}

self.df.read() acquires a parking_lot read lock (parking_lot is a faster, drop-in replacement for std::sync::RwLock), returning a guard that derefs to &DataFrame.

Step 3: polars-core — crates/polars-core/src/frame/dataframe.rs:226–228 

#[inline]
pub fn height(&self) -> usize {
    self.height
}

It's a field read. The DataFrame caches its height as a plain usize. 

  df.height                         (Python)
    │
    ▼
  self._df.height()                 (Python → PyO3 call)
    │
    ▼
  PyDataFrame::height(&self)        (Rust, polars-python)
    │  self.df.read().height()
    │       │
    │       ▼
    │  RwLock::read() → &DataFrame
    │       │
    │       ▼
    │  DataFrame::height(&self)     (Rust, polars-core)
    │       │
    │       ▼
    │  return self.height (usize)
    │
    ▼
  usize → Python int               (PyO3 auto-conversion)

This is the pattern for every simple accessor. The cost is one Python-to-Rust function call (via PyO3's generated FFI), one RwLock read-lock acquisition, one field access, and PyO3's automatic conversion of usize to a Python int. No data copying, no GIL release needed.

What Python users actually hold

That trace bottoms out in PyDataFrame and DataFrame. Here's what those types actually are — and the design pattern that makes the whole boundary work.

Four #[pyclass] structs form the boundary layer. PyDataFrame, PySeries, and PyLazyFrame each wrap their Rust counterpart for Python access — a LazyFrame represents a query plan that hasn't executed yet. PyExpr wraps an expression node: a description of an operation, not data itself.

PyDataFrame

}

PyLazyFrame

PyExpr — the odd one out

crates/polars-python/src/expr/mod.rs:37–42

#[pyclass] // Not marked frozen for pickling, but that's the only &mut self method.
#[repr(transparent)]
#[derive(Clone)]
pub struct PyExpr {
    pub inner: Expr,
}

PyExpr is not frozen and does not use RwLock. The comment explains why: pickling — Python's serialization mechanism for converting objects to bytes for storage or transfer — requires &mut self, and that's the only mutation path. Since Expr is a pure data structure (an AST node), there's no interior-mutability concern — cloning is cheap and safe.

The frozen + RwLock pattern

The other three structs share the same design:

  1. #[pyclass(frozen)] tells PyO3 that Python will never see a &mut self reference. Python can hold multiple references to the same object without violating Rust's aliasing rules. PyO3 doesn't need runtime borrow-checking — every Python method receives &self.
  2. RwLock<T> (from parking_lot, not std) provides interior mutability within &self. Methods that read call self.df.read() for a shared lock. Methods that mutate call self.df.write() for exclusive access.
  3. #[repr(transparent)] guarantees that PyDataFrame has exactly the same memory layout as RwLock<DataFrame>. This is exploited in PyExpr's ToExprs and ToPyExprs impls, which use unsafe pointer casts to convert Vec<PyExpr> to Vec<Expr> at zero cost (expr/mod.rs:54–66).
  Python land                  PyO3 boundary            Rust land
  ──────────                   ─────────────            ──────────
  polars.DataFrame ──holds──> PyDataFrame
        ._df                    │
                                ├── RwLock<DataFrame>
                                │     .read()  ──────> &DataFrame
                                │     .write() ──────> &mut DataFrame
                                │
                                └── DataFrame (polars-core)
                                      ├── height: usize
                                      ├── columns: Vec<Column>
                                      └── cached_schema: OnceLock<SchemaRef>

Every new #[pymethods] function you add to PyDataFrame, PySeries, or PyLazyFrame will receive &self. You access the inner data through self.df.read() (or self.series.read(), self.ldf.read()). This is the first thing you'll type in any new binding function.

How an expression is built — pl.col("x")

Expressions are the heart of Polars' API. An expression is a description of an operation: "take column x," "sum it," "filter where it's greater than 5." Rather than executing immediately, Polars builds a tree of these descriptions first, then executes the whole tree later. When a user writes pl.col("x"), they're creating the leaf node of that tree — "refer to the column named x." Each subsequent method call wraps the existing tree in a new node, building up an abstract syntax tree.

Step 1: pl.col is a Col instance — py-polars/src/polars/functions/col.py:432 

col: Col = Col()

pl.col isn't a function — it's an instance of the Col class. When you call pl.col("x"), Python invokes Col.__call__.

Step 2: Col.__call__ dispatches — col.py:210–363

For a simple string argument, __call__ delegates to _create_col(name).

Step 3: _create_col for strings — col.py:70–71 

if isinstance(name, str):
    return wrap_expr(plr.col(name))

plr is polars._plr, the compiled Rust shared library that gets loaded at import polars time — the same extension module described in the opening. plr.col(name) is a #[pyfunction] call that crosses the PyO3 boundary.

Step 4: Rust col() function — crates/polars-python/src/functions/lazy.rs:113–116 

#[pyfunction]
pub fn col(name: &str) -> PyExpr {
    dsl::col(name).into()
}

PyO3 extracts the Python str to a Rust &str, then calls into polars-plan.

Step 5: The Expr is created — crates/polars-plan/src/dsl/functions/selectors.rs:27–37 

pub fn col<S>(name: S) -> Expr
where
    S: Into<PlSmallStr>,
{
    let name = name.into();
    match name.as_str() {
        "*" => all().as_expr(),
        n if is_regex_projection(n) => Expr::Selector(Selector::Matches(name)),
        _ => Expr::Column(name),
    }
}

For "x", this produces Expr::Column(PlSmallStr::from("x")) — a single enum variant holding a small string.

Step 6: Wrapping back to Python — py-polars/src/polars/_utils/wrap.py 

def wrap_expr(pyexpr: PyExpr) -> Expr:
    return pl.Expr._from_pyexpr(pyexpr)

The Rust PyExpr (wrapping Expr::Column("x")) becomes a Python polars.Expr object. The user sees a Python Expr whose _pyexpr attribute holds the Rust PyExpr.

Step 7: Chaining — pl.col("x").sum()

When the user calls .sum(), the Python Expr.sum() method (py-polars/src/polars/expr/expr.py:3523) calls self._pyexpr.sum(), which hits: 

// crates/polars-python/src/expr/general.rs:151–153
fn sum(&self) -> Self {
    self.inner.clone().sum().into()
}

This calls Expr::sum() from crates/polars-plan/src/dsl/statistics.rs:71–73: 

pub fn sum(self) -> Self {
    AggExpr::Sum(Arc::new(self)).into()
}

So pl.col("x").sum() produces Expr::Agg(AggExpr::Sum(Arc::new(Expr::Column("x")))). A tree of enum variants. No computation yet. 

  pl.col("x")                             Python
    │
    ▼
  Col.__call__("x") → _create_col("x")    Python
    │
    ▼
  plr.col("x")                            PyO3 boundary crossing
    │
    ▼
  dsl::col("x")                           polars-plan
    │
    ▼
  Expr::Column(PlSmallStr("x"))           AST node created
    │
    ▼
  PyExpr { inner: Expr::Column("x") }     wrapped for Python
    │
    ▼
  wrap_expr() → polars.Expr               Python Expr object
    │
    ▼  .sum()
  PyExpr::sum() → Expr::Agg(Sum(Column("x")))

Adding a new expression function means: (1) adding a variant or FunctionExpr case in polars-plan, (2) adding a method to PyExpr in crates/polars-python/src/expr/general.rs (or a sub-module), and (3) adding the Python wrapper in py-polars/src/polars/expr/expr.py. Each .method() call crosses the PyO3 boundary once, clones the Expr tree, wraps it in a new node, and returns.

Eager vs lazy — where the boundary matters

Polars has two execution modes. Eager means compute immediately and return a result: df.select() gives you a new DataFrame right away. Lazy means record the operations as a plan, optimize that plan, then compute when you finally call .collect().

The most surprising thing about these two modes: eager operations don't have a separate implementation. df.select() calls self.lazy().select().collect() internally. All eager DataFrame methods go through the lazy engine with a reduced optimization set. Here's the full trace.

Eager: df.select(pl.col("x").sum())

py-polars/src/polars/dataframe/frame.py: 

def select(self, *exprs, **named_exprs):
    from polars.lazyframe.opt_flags import QueryOptFlags
    return (
        self.lazy()
        .select(*exprs, **named_exprs)
        .collect(optimizations=QueryOptFlags._eager())
    )

Eager select is secretly lazy. It calls self.lazy(), builds a lazy plan, then immediately .collect()s it. The _eager() flag (py-polars/src/polars/lazyframe/opt_flags.py:142–148) disables most optimizations but keeps simplify_expression and sets an eager flag.

The full boundary-crossing count for df.select(pl.col("x").sum()):

  1. pl.col("x") — crosses once (Python → plr.col)
  2. .sum() — crosses once (_pyexpr.sum())
  3. df.lazy() — crosses once (_df.lazy() → PyDataFrame::lazy() at general.rs:453)
  4. .select([pyexpr]) — crosses once (_ldf.select() at general.rs:859)
  5. .with_optimizations(flags) — crosses once (general.rs:541)
  6. .collect(engine, callback) — crosses once (general.rs:566) — GIL is released here via py.enter_polars_df()

Total: 6 PyO3 boundary crossings, with the GIL released during actual computation.

Lazy: df.lazy().select(pl.col("x").sum()).collect()

The user writes it explicitly, but the call sequence is nearly identical:

  1. pl.col("x") — crosses once
  2. .sum() — crosses once
  3. df.lazy() — crosses once
  4. .select([pyexpr]) — crosses once
  5. .collect() — calls _ldf.with_optimizations(flags) (crosses once), then ldf.collect(engine, callback) (crosses once, GIL released)

Total: 6 PyO3 boundary crossings — the same number.

So what's different?

The difference is in the OptFlags. Explicit lazy uses the default QueryOptFlags, enabling the full optimizer suite: projection pushdown (skip columns the query never reads), predicate pushdown (filter rows before loading all the data), common subplan elimination, and join collapsing. Eager mode disables all of these except simplify_expression.

The optimizer runs inside the collect() call, after the GIL is released. ldf.collect_with_engine(engine) (crates/polars-python/src/lazyframe/general.rs:579) delegates to polars-lazy, which runs the optimizer and then the execution engine — all in pure Rust, without the GIL. 

  EAGER PATH:                         LAZY PATH:
  df.select(pl.col("x").sum())        df.lazy().select(pl.col("x").sum()).collect()
    │                                   │
    ▼                                   ▼
  self.lazy()  ──────────────────────  df.lazy()
    │                                   │
    ▼                                   ▼
  .select(exprs)                       .select(exprs)
    │                                   │
    ▼                                   ▼
  .collect(                            .collect(
    optimizations=_eager()               optimizations=DEFAULT  ← full optimizer
  )                                    )
    │                                   │
    ▼                                   ▼
  with_optimizations(flags)            with_optimizations(flags)
    │                                   │
    ▼                                   ▼
  ldf.collect(engine, callback)        ldf.collect(engine, callback)
    │                                   │
    ▼                                   ▼
  py.enter_polars_df(|| {              py.enter_polars_df(|| {
    ldf.collect_with_engine(e)           ldf.collect_with_engine(e)
  })  ← GIL released here             })  ← GIL released here

The enter_polars_df helper (crates/polars-python/src/utils.rs:105–111) is the canonical pattern for releasing the GIL. It calls self.detach(|| ...) (PyO3's GIL-release mechanism), wraps the closure in catch_unwind for panic safety, handles KeyboardInterrupt via a timeout, and converts the result to PyDataFrame. Any new binding that does significant computation should use this pattern.

Where data actually crosses

The previous sections covered metadata and AST nodes crossing the boundary. This one is about data — the actual column buffers that hold millions of values.

The key concept here is zero-copy: Rust reading the same memory buffer that Python or NumPy allocated, without copying it to a new location. Why it matters is straightforward — copying a billion numbers takes time and doubles memory usage. Every data-crossing path in Polars is designed to avoid that copy when possible.

Python → Rust: Building a Series from a Python list

When a user writes pl.Series("x", [1, 2, 3]), the Python constructor eventually calls one of several PySeries static methods depending on the inferred type.

Fast path: NumPy arrays — crates/polars-python/src/series/construction.rs:22–55

For typed NumPy arrays, a macro generates per-type constructors (new_i8, new_i16, ..., new_u64). These call numpy_array_to_arrow(): 

fn numpy_array_to_arrow<T: Element + NativeType>(
    array: &Bound<PyArray1<T>>
) -> PrimitiveArray<T> {
    let owner = array.clone().unbind();
    let ro = array.readonly();
    let vals = ro.as_slice().unwrap();
    unsafe {
        let storage = SharedStorage::from_slice_with_owner(vals, owner);
        let buffer = Buffer::from_storage(storage);
        PrimitiveArray::new_unchecked(T::PRIMITIVE.into(), buffer, None)
    }
}

This is zero-copy. It takes the NumPy array's memory buffer, wraps it in SharedStorage — part of Polars' Arrow implementation, which stores data in Apache Arrow's columnar memory format — and creates an Arrow PrimitiveArray pointing directly at the same memory. Arrow is a standardized in-memory columnar layout, and because Polars uses it internally, zero-copy exchange with other Arrow-aware tools is possible: everyone agrees on how data is laid out in memory. The SharedStorage retains a Python reference to prevent GC. No data is copied.

Slow path: Python lists — construction.rs:248–275

For plain Python lists, new_from_any_values iterates through the Python sequence and calls py_object_to_any_value on each element. This is inherently expensive — O(n) Python object inspections. 

fn new_from_any_values(name: &str, values: &Bound<PyAny>, strict: bool) -> PyResult<Self> {
    let any_values_result = values
        .try_iter()?
        .map(|v| py_object_to_any_value(&(v?).as_borrowed(), strict, true))
        .collect::<PyResult<Vec<AnyValue>>>();
    // ...
    let s = Series::from_any_values(name.into(), avs.as_slice(), strict);
    // ...
}

construction.rs:295–312 also has a dedicated string path that uses a StringChunkedBuilder, iterating and extracting Cow<str> from each Python object.

Python → Rust: DataFrames

dict_to_pydf (py-polars/src/polars/_utils/construction/dataframe.py:76) is the entry point for pl.DataFrame({"a": [1,2,3]}). It processes each column value through the Series construction pipeline above, then assembles them into a PyDataFrame. For 3+ large NumPy arrays with nan_to_null=True, it uses multiprocessing.dummy (thread pool) to parallelize the conversion (dataframe.py:112–145).

Rust → Python: .to_list()

The reverse direction is in crates/polars-python/src/series/export.rs:17. The to_list method is a big match on series.dtype():

    let series = &self.series.read();
    fn to_list_recursive<'py>(py: Python<'py>, series: &Series) -> PyResult<Bound<'py, PyAny>> {
        let pylist = match series.dtype() {
            DataType::Boolean => PyList::new(py, series.bool()...)?,
            DataType::UInt8   => PyList::new(py, series.u8()...)?,
            // ... every type handled explicitly
            DataType::List(_) => { /* recursive case for nested lists */ },

This always copies data. Every Rust value becomes a Python object. For a million-row integer column, that's a million Python int objects allocated on the Python heap.

Rust → Python: Arrow PyCapsule (zero-copy export)

The modern zero-copy path uses Arrow's C Data Interface via PyCapsules — Python's mechanism for passing raw pointers between extension modules. __arrow_c_stream__ (export.rs:169–175) exports a Series as an Arrow stream capsule: 

fn __arrow_c_stream__<'py>(&self, py: Python<'py>, ...) -> PyResult<Bound<'py, PyCapsule>> {
    series_to_stream(&self.series.read(), py)
}

This is what PyArrow and other Arrow-aware libraries use to receive Polars data without copying. 

  PYTHON → RUST (data ingestion):

  pl.Series([1,2,3])
    │
    ├── NumPy array? ──YES──> numpy_array_to_arrow()         ZERO-COPY
    │                           SharedStorage wraps buffer
    │
    └── Python list? ──YES──> new_from_any_values()           O(n) COPY
                                py_object_to_any_value per element

  RUST → PYTHON (data export):

  series.to_list()
    │
    └── match on dtype ──> PyList::new(py, ...) per type      O(n) COPY

  series.__arrow_c_stream__()
    │
    └── series_to_stream() → PyCapsule                        ZERO-COPY

If you're adding a new data-ingestion path, the question is always: can we get an Arrow buffer without copying? If the source is NumPy or Arrow-compatible, use SharedStorage or PyCapsule import. If it's a Python iterable of Python objects, you're stuck with O(n) conversion — try to get the dtype early so you can use a typed builder (like StringChunkedBuilder) rather than the AnyValue fallback.

The plugin boundary

The pyo3-polars crate (at pyo3-polars/ in the repo) provides a completely different mechanism from the internal crates/polars-python/ bindings. A plugin is a separately compiled shared library — it can't share Rust types directly with the main Polars binary because they're different compilation units with potentially different type layouts. The Arrow C Data Interface solves this by providing a C-level struct layout that both sides can agree on, regardless of how each was compiled.

Internal bindings: crates/polars-python/

The internal PyDataFrame, PySeries, etc. are #[pyclass] structs compiled into the polars Python extension module. They share the same process, the same allocator, and the same Rust types. When PyDataFrame calls self.df.read(), it's a direct pointer dereference to a DataFrame on the same heap.

Plugin bindings: pyo3-polars/pyo3-polars/src/types.rs

The plugin crate defines its own PySeries and PyDataFrame:

// pyo3-polars/pyo3-polars/src/types.rs:36–43
#[repr(transparent)]
pub struct PySeries(pub Series);

#[repr(transparent)]
pub struct PyDataFrame(pub DataFrame);

No RwLock. No #[pyclass]. These are plain newtypes in a different shared library from the main Polars module.

How data crosses between the two libraries

When a plugin receives a PySeries, the FromPyObject impl (types.rs:176–186) does something clever: 

impl<'a, 'py> FromPyObject<'a, 'py> for PySeries {
    fn extract(ob: Borrowed<'a, 'py, PyAny>) -> PyResult<Self> {
        let mut export = SeriesExport::empty();
        ob.getattr("_s")?
            .call_method1("_export", ((&raw mut export).addr(),))?;
        let series = unsafe { import_series(export).map_err(PyPolarsErr::from)? };
        Ok(PySeries(series))
    }
}

It calls _export on the internal PySeries (from crates/polars-python/), passing a raw pointer to a SeriesExport struct. The internal side writes Arrow C Data Interface structs into that memory (crates/polars-python/src/series/export.rs:177–182): 

pub fn _export(&self, _py: Python<'_>, location: usize) {
    let export = polars_ffi::version_0::export_series(&self.series.read());
    unsafe {
        (location as *mut SeriesExport).write(export);
    }
}

SeriesExport itself (crates/polars-ffi/src/version_0.rs) is a #[repr(C)] struct — using the C calling convention, the common interface that different languages and separately compiled libraries can agree on — containing Arrow schema and array pointers: 

#[repr(C)]
pub struct SeriesExport {
    field: *mut ArrowSchema,
    arrays: *mut *mut ArrowArray,
    len: usize,
    release: Option<unsafe extern "C" fn(arg1: *mut SeriesExport)>,
    private_data: *mut std::os::raw::c_void,
}

The data doesn't get copied. The plugin library receives raw pointers to the same Arrow buffers that live in the main Polars process. Cross-library zero-copy.

How #[polars_expr] works

The polars_expr proc macro (pyo3-polars/pyo3-polars-derive/src/lib.rs:263–285) generates two extern "C" functions for each annotated Rust function:

  1. _polars_plugin_{name} — receives SeriesExport pointers (inputs) and writes a SeriesExport (output). This is the computation function.
  2. _polars_plugin_field_{name} — receives ArrowSchema pointers (input field metadata) and returns an ArrowSchema (output type). This is the type-inference function used during query planning.

The generated code for the computation function (lib.rs:136–163): 

#[no_mangle]
pub unsafe extern "C" fn _polars_plugin_my_func(
    e: *mut SeriesExport,
    input_len: usize,
    kwargs_ptr: *const u8,
    kwargs_len: usize,
    return_value: *mut SeriesExport,
    context: *mut CallerContext,
) {
    let panic_result = std::panic::catch_unwind(move || {
        let inputs = import_series_buffer(e, input_len).unwrap();
        // ... call user function ...
        *return_value = export_series(&out);
    });
}

Why this bypasses the GIL

These are extern "C" functions loaded via dlopen / dlsym — the OS-level functions for loading shared libraries at runtime and looking up symbols in them. Polars' engine calls them directly from Rust, not through Python. Series data arrives via C-ABI pointers, computation happens in pure Rust, and results go back via C-ABI pointers. Python is never involved in the hot path. The GIL doesn't need to be held because no Python objects are created, accessed, or modified. 

  INTERNAL BINDINGS (crates/polars-python/):

  Python obj ──PyO3 #[pymethods]──> &self.df.read() ──> DataFrame
                                    (same process, same heap)

  PLUGIN BINDINGS (pyo3-polars/):

  Python obj ──.getattr("_s")._export(ptr)──> SeriesExport (Arrow C ABI)
    │                                              │
    │  (pointer handoff, zero copy)                │
    │                                              ▼
    │                                         import_series()
    │                                              │
    │                                              ▼
    │                                         Plugin's own Series
    │                                              │
    │                                    (pure Rust, no GIL, no Python)
    │                                              │
    │                                              ▼
    │                                         export_series(&result)
    │                                              │
    ▼                                              ▼
  Result imported back               SeriesExport written to return_value

If you're building an expression plugin, you never touch PyO3 #[pymethods] at all. You write fn my_func(inputs: &[Series]) -> PolarsResult<Series>, annotate it with #[polars_expr(output_type=...)], and compile to a cdylib (a Rust crate type that produces a C-compatible shared library). The macro handles all the FFI plumbing. Your function receives real Series values and returns a real Series — the C-ABI boundary is invisible.

Key takeaways for contributors

  1. The four #[pyclass] structs are your entry points. PyDataFrame, PySeries, PyLazyFrame use frozen + RwLock. PyExpr is the exception: no lock, just a cloneable AST node.
  2. Eager operations go through lazy internally. df.select() calls self.lazy().select().collect(). If you're optimizing an eager path, you're actually optimizing the lazy path with limited optimization flags.
  3. Expression building is cheap; execution is where the work happens. Every .method() on an Expr is one PyO3 crossing that clones an enum and wraps it in a new node. The real computation happens inside enter_polars_df, after the GIL is released.
  4. Data conversion is the expensive boundary. Python list → Series is O(n) with per-element inspection. NumPy → Series is zero-copy via SharedStorage. Arrow PyCapsule exchange is zero-copy in both directions.
  5. Plugins bypass everything. The #[polars_expr] system uses C-ABI FFI, not PyO3 methods. Data arrives as Arrow buffers, computation runs in pure Rust, results return as Arrow buffers. If you're writing a plugin, you never interact with the PyO3 layer at all.
