Rust Design Challenges — #14 of 27

Two places read this data. A third place writes it. Structure that — Solution

March 2026 · Part B: The Solution

The answer: A

Use Arc<RwLock<Settings>>.

RwLock mirrors Rust's borrow rules at runtime: many readers or one writer, never both. When two threads call .read(), they both get access simultaneously — no blocking. When the writer calls .write(), it waits for all readers to finish, then gets exclusive access.

This matches the access pattern perfectly: reads are frequent and concurrent, writes are rare and exclusive. A Mutex would force readers to queue behind each other, which is unnecessary when all they're doing is reading. RwLock gives you the concurrency the problem demands.

Arc is needed because the RwLock is shared across threads. Arc provides thread-safe shared ownership; RwLock provides thread-safe reader/writer access. Together, they give you exactly the semantics of "many readers, one writer, across threads."

The solution

use std::collections::HashMap;
use std::sync::{Arc, RwLock};
use std::thread;

struct Settings {
    values: HashMap<String, String>,
}

impl Settings {
    fn new() -> Self {
        Settings { values: HashMap::new() }
    }

    fn get(&self, key: &str) -> Option<String> {
        self.values.get(key).cloned()
    }

    fn set(&mut self, key: &str, value: &str) {
        self.values.insert(key.into(), value.into());
    }
}

fn main() {
    let settings = Arc::new(RwLock::new(Settings::new()));

    // Writer thread
    {
        let s = Arc::clone(&settings);
        thread::spawn(move || {
            s.write().unwrap().set("theme", "dark");
        });
    }

    // Two reader threads — can read concurrently
    let mut handles = vec![];
    for i in 0..2 {
        let s = Arc::clone(&settings);
        handles.push(thread::spawn(move || {
            let guard = s.read().unwrap();
            println!("reader {}: theme={:?}", i, guard.get("theme"));
        }));
    }

    for h in handles {
        h.join().unwrap();
    }
}

Why the other options fail

B. Arc<Mutex<>> — works, but readers block each other. If two threads both want to read settings, one waits for the other to release the lock. With a read-heavy workload, this kills concurrency.

C. Clone for each reader — each reader gets a snapshot, which goes stale immediately. Changes from the writer never reach the readers. If you need readers to see updates, cloning is the wrong answer.

D. Cell — Cell is for Copy types only, and it's not Sync — you can't share it across threads. Settings contains a HashMap, which isn't Copy.

Many readers, one writer, across threads: that's RwLock's exact job.
