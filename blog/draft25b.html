This tree needs an iterator. Write one — Solution

March 2026 · Part B: The Solution

The answer: A

Use an explicit stack of references in an iterator struct.

Recursion is the natural way to walk a tree, but iterators need to pause and resume. Rust doesn't have stable generators or yield, so you convert the recursive algorithm into an iterative one using an explicit stack. Each call to next() pops from the stack, processes the node, pushes children, and returns when it finds a leaf.

The iterator struct holds Vec<&'a Expr> — a stack of references into the tree. The lifetime 'a ties the iterator to the tree it borrows from, ensuring the tree isn't dropped while iteration is in progress.

This is the standard way to write tree iterators in Rust: stack-based, borrowing, lazy. Each next() call does only the work needed to find the next value.

The solution

enum Expr {
    Num(i64),
    Add(Box<Expr>, Box<Expr>),
    Mul(Box<Expr>, Box<Expr>),
}

struct LeafIter<'a> {
    stack: Vec<&'a Expr>,
}

impl<'a> LeafIter<'a> {
    fn new(root: &'a Expr) -> Self {
        LeafIter { stack: vec![root] }
    }
}

impl<'a> Iterator for LeafIter<'a> {
    type Item = i64;

    fn next(&mut self) -> Option<Self::Item> {
        while let Some(node) = self.stack.pop() {
            match node {
                Expr::Num(n) => return Some(*n),
                Expr::Add(a, b) | Expr::Mul(a, b) => {
                    // Push right first so left is processed first (stack is LIFO)
                    self.stack.push(b);
                    self.stack.push(a);
                }
            }
        }
        None
    }
}

fn main() {
    // (2 + 3) * 4
    let tree = Expr::Mul(
        Box::new(Expr::Add(
            Box::new(Expr::Num(2)),
            Box::new(Expr::Num(3)),
        )),
        Box::new(Expr::Num(4)),
    );

    let leaves: Vec<i64> = LeafIter::new(&tree).collect();
    println!("leaves: {:?}", leaves); // [2, 3, 4]
}

Why the other options fail

B. Collect into Vec first — works but isn't lazy. It traverses the entire tree upfront and allocates a Vec to hold all values. If you only need the first two leaves, you've done unnecessary work. An iterator should be pay-as-you-go.

C. Recursion with yield — Rust has no stable yield. Generators are experimental. The stack-based approach is the stable equivalent of what generators would give you.

D. Consuming IntoIterator — moves nodes out of the tree during iteration. After iterating, the tree is gone. If you want to iterate and then do something else with the tree, you can't. Borrowing preserves the tree.

To iterate a tree in Rust: convert recursion to an explicit stack of references.
