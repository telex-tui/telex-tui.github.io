This function returns a reference. Why won't it compile? — Solution

March 2026 · Part B: The Solution

The answer: A

Return an owned String. Accept the allocation.

The formatted name doesn't exist before the function runs. format! creates a new String on the heap, owned by the local variable. When the function returns, that local is dropped. A reference to it would dangle. There is no way to return a reference to something the function just created — the thing won't exist after the return.

This is a fundamental constraint, not a limitation you can work around with annotations. A reference is a borrow of something that already exists. If no one else owns the data, you must return it by value.

The desire to "avoid allocation" is reasonable, but in this case the allocation is inherent to the operation. Combining two strings into one requires new memory. The right answer is to return the owned String and let the caller decide what to do with it. If the caller needs a &str, they can borrow the result after receiving it.

The solution

struct User {
    first: String,
    last: String,
}

impl User {
    fn new(first: &str, last: &str) -> Self {
        User { first: first.into(), last: last.into() }
    }
}

// Returns owned String — the formatted name is new data
fn display_name(user: &User) -> String {
    format!("{} {}", user.first, user.last)
}

fn main() {
    let user = User::new("Alice", "Smith");
    let name = display_name(&user);
    // Caller owns the String, can borrow it as &str
    println!("{}", &name);
}

Why the other options fail

B. Lifetime parameter with &'a str — the lifetime parameter connects the output to the input. But the formatted string isn't part of the input — it's newly created. There's nothing in the User to point to. A lifetime annotation can't create ownership; it can only describe existing relationships.

C. Box::leak — turns the String into a &'static str by leaking the memory permanently. The memory is never freed. Do this in a loop and you've got an unbounded memory leak.

D. Store as a field on User — precomputing and caching the name is a valid optimisation if you call this frequently. But it changes the data model — now User has to be updated whenever first or last changes. The question is about the function return, and the direct answer is: return the owned value.

If a function creates new data, it must return it by value — you can't return a reference to something that won't exist after the return.
