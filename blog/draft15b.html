This iterator needs to yield borrowed data. Make it compile — Solution

March 2026 · Part B: The Solution

The answer: A

Create an iterator struct with a lifetime parameter that borrows the word list.

When an iterator yields references, those references point into data owned by something else. The compiler needs to know that the data won't be dropped while the iterator is alive. That's what the lifetime parameter does — it ties the iterator's lifetime to the lifetime of the data it borrows.

The iterator struct holds a &'a [String] — a slice borrowed from the WordList. Each call to next() returns an Option<&'a str>. The lifetime 'a tells the compiler: "these references are valid as long as the WordList they came from is alive."

This is exactly how the standard library's slice iterator works. You're not inventing a pattern — you're applying the one Rust is built around.

The solution

struct WordList {
    words: Vec<String>,
}

impl WordList {
    fn new(words: Vec<String>) -> Self {
        WordList { words }
    }

    fn iter(&self) -> WordIter {
        WordIter {
            words: &self.words,
            pos: 0,
        }
    }
}

// The lifetime 'a ties this iterator to the WordList it borrows from
struct WordIter<'a> {
    words: &'a [String],
    pos: usize,
}

impl<'a> Iterator for WordIter<'a> {
    type Item = &'a str;

    fn next(&mut self) -> Option<Self::Item> {
        if self.pos < self.words.len() {
            let word = &self.words[self.pos];
            self.pos += 1;
            Some(word.as_str())
        } else {
            None
        }
    }
}

fn main() {
    let list = WordList::new(vec![
        "hello".into(),
        "world".into(),
        "rust".into(),
    ]);

    for word in list.iter() {
        println!("{}", word);
    }
}

Why the other options fail

B. Clone each string — compiles, but defeats the purpose of an iterator over borrowed data. You're allocating on every iteration for no reason. If the caller wanted owned strings, they could call .cloned() on the iterator.

C. Leak with 'static — Box::leak gives you &'static str but permanently leaks the memory. This isn't lazy iteration — it's a memory leak pretending to be an API.

D. No lifetime parameter — the compiler can't verify that the returned &str references are valid without knowing their lifetime. The struct holds a reference, and references on structs always need a lifetime parameter. No lifetime, no compilation.

An iterator that borrows data needs a lifetime parameter tying it to the data's owner.
