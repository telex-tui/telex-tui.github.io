Fan out to three tasks. Collect all results. Handle one failure — Solution

March 2026 · Part B: The Solution

The answer: A

Use tokio::try_join!.

try_join! polls all three futures concurrently on the same task. If all succeed, it returns a tuple of the results. If any one returns Err, it returns that error immediately and cancels the remaining futures.

This is exactly the semantics you want: "I need all three results to proceed. If any fails, there's no point waiting for the rest." It's the concurrent, fallible version of sequential awaits with ?.

The three futures run on the same task (no spawning, no extra threads). They're polled cooperatively — when one is waiting on I/O, the others make progress. It's concurrent but not parallel. For actual parallelism, you'd spawn separate tasks and try_join! their join handles.

The solution

async fn fetch_users() -> Result<Vec<String>, String> {
    Ok(vec!["Alice".into(), "Bob".into()])
}

async fn fetch_orders() -> Result<Vec<String>, String> {
    Ok(vec!["order-1".into(), "order-2".into()])
}

async fn fetch_products() -> Result<Vec<String>, String> {
    Ok(vec!["widget".into(), "gadget".into()])
}

#[tokio::main]
async fn main() {
    // All three run concurrently; first error short-circuits
    match tokio::try_join!(fetch_users(), fetch_orders(), fetch_products()) {
        Ok((users, orders, products)) => {
            println!("users: {:?}", users);
            println!("orders: {:?}", orders);
            println!("products: {:?}", products);
        }
        Err(e) => {
            println!("one fetch failed: {}", e);
        }
    }
}

Why the other options fail

B. Spawn and await sequentially — awaiting one after another runs them in sequence: fetch users, wait, fetch orders, wait, fetch products, wait. Three round trips instead of one. The whole point is concurrency.

C. join! then check errors — join! waits for all three to complete, even if the first one fails immediately. You pay the full latency of all three even when the answer is already "fail." try_join! short-circuits.

D. select! — returns the result of whichever future completes first and cancels the rest. You'd get one result, not all three. select! is for "first one wins" semantics, not "all must succeed."

Need all results, fail on any error: that's try_join!.
