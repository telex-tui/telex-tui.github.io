The GIL Is Just a Mutex: A Rust Developer's Guide to PyO3 and Python's Concurrency Model

March 2026

Draft вҖ” work in progress, not for publication.

If you're writing Rust that gets called from Python via PyO3, you need a mental model of the GIL. Not the "GIL bad, Python slow" meme вҖ” an actual working model of what's happening, so you can design your FFI boundary well.

Here's the good news: if you've ever used a mutex, you already understand it.

What the GIL Actually Is

The Global Interpreter Lock is a pthread_mutex (or the Windows equivalent) inside CPython. It protects the interpreter's shared state: reference counts, the object graph, the memory allocator. Only the thread holding the GIL can execute Python bytecode or touch Python objects.

That's it. There's no special scheduler, no memory barrier dance beyond what the mutex provides. It's a lock protecting a data structure вҖ” the data structure just happens to be an entire language runtime.

This means that CPU-bound Python code running across multiple threads will serialise. The threads take turns, one at a time, on the mutex. Multiple cores sit idle. This is the part everyone complains about.

But it also means the GIL can be released. And that's where it gets interesting.

The PyO3 Paradigm

When Python calls into Rust via PyO3, there's a single pattern that matters: 

Python (GIL held)
  вҶ’ convert Python objects to Rust-owned data
    вҶ’ release the GIL
      вҶ’ Rust does whatever it wants
    вҶ’ reacquire the GIL
  вҶ’ convert Rust results back to Python objects
Python (GIL held)

In code, the release point looks like this:

py.allow_threads(|| {
    // GIL is released
    // Full multi-core parallelism available
    // But you CANNOT touch any Python objects
    heavy_computation(&rust_data)
})

Under the hood, allow_threads calls CPython's PyEval_SaveThread(), which does roughly: 

PyThreadState* PyEval_SaveThread(void) {
    PyThreadState *tstate = current_thread_state;
    current_thread_state = NULL;
    drop_gil(tstate);   // unlock the mutex
    return tstate;
}

It unlocks the mutex. That's the whole trick.

The contract is explicit: you give up the GIL, you gain parallelism, you lose access to anything Python. No PyObject, no PyList, no calling back into the interpreter. Rust's type system enforces this вҖ” the allow_threads closure can't capture any Python<'py> token or Bound<'py, T> references. The compiler won't let you cheat.

Here's where the model gets genuinely powerful. When Rust releases the GIL, the GIL is free. Other Python threads can grab it.

The Python threads take turns with the GIL to dispatch and collect. The Rust computations run simultaneously across all available cores. The GIL is only contended during the brief boundary crossings, not during the actual work.

This is how libraries like Polars achieve their performance. The Rust engine runs a Rayon thread pool across all cores while Python idles, GIL available, doing nothing until results come back.

Python objects live on CPython's heap, managed by its allocator and refcounting. Rust objects live in Rust's own memory. The two worlds genuinely don't overlap unless you bridge them.

The data copy at the boundary is the design decision that shapes everything. You have a spectrum:

Deep copy вҖ” convert a Python list into a Vec<f64>. Rust owns it completely, clean separation, simple ownership. This is the safe default.

Zero-copy вҖ” use the buffer protocol or Arrow format so Rust gets a pointer into memory Python also knows about. Faster, but now you need to ensure Python doesn't free the buffer while Rust is working with it.

Libraries like Polars sit at the zero-copy end: data lives in Rust-owned Arrow buffers and only crosses the boundary when Python actually needs to inspect values. The bridge layer exists to minimise crossings, not to copy data back and forth.

The Boundary Is the Architecture

The quality of a PyO3 bridge comes down to how it manages the boundary crossings. Every crossing means:

  1. Acquiring or releasing the GIL (mutex lock/unlock)
  2. Converting data between Python and Rust representations
  3. A potential serialisation point where parallelism stalls

A poorly designed boundary forces too many acquire/release cycles or too much copying. A well-designed one keeps Rust running free for as long as possible: 

    Bad: many small crossings         Good: few large crossings

  Python вҶҗвҶ’ Rust вҶҗвҶ’ Python вҶҗвҶ’ Rust   Python вҶ’ Rust в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖвҶ’ Python
  Python вҶҗвҶ’ Rust вҶҗвҶ’ Python вҶҗвҶ’ Rust
  Python вҶҗвҶ’ Rust вҶҗвҶ’ Python вҶҗвҶ’ Rust   (GIL released once, Rust does
  (GIL thrashing, per-row overhead)    everything, returns final result)

This is why Polars uses lazy evaluation. Instead of crossing the boundary for each operation, Python builds up a query plan as a lightweight description. One boundary crossing sends the whole plan to Rust, the GIL gets released once, Rust executes the full plan across all cores, and one crossing brings results back.

The Mental Model

If you're coming from C, here's the mapping:

C concept GIL equivalent
pthread_mutex_lock() Acquiring the GIL
pthread_mutex_unlock() py.allow_threads()
Shared data protected by the mutex Python objects on CPython's heap
Thread-local data, no mutex needed Rust-owned data after GIL release
Worker thread pool behind a mutex-protected queue PyO3 dispatch pattern

The only unusual thing is that the mutex protects an entire language runtime rather than a data structure. But the discipline is identical: hold the lock to access shared state, release it to do independent work, reacquire it to publish results.

PyO3's allow_threads makes this Rust-native by encoding the discipline in the type system. You can't accidentally touch Python objects with the GIL released, because the compiler won't let the closure capture them. It's a compile-time version of what C programmers enforce by convention.

That's Rust's recurring trick: taking patterns that experienced C programmers already follow and making the compiler enforce them. The GIL boundary in PyO3 is one of the cleanest examples.
