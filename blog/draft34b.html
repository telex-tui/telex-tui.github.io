Two async tasks need to coordinate on a single result — Solution

March 2026 · Part B: The Solution

The answer: A

Use tokio::sync::oneshot.

A oneshot channel is purpose-built for this pattern: one sender, one receiver, one message. The producer sends the result. The consumer .awaits the receiver. The consumer is suspended until the value arrives — no polling, no wasted CPU cycles.

The types enforce the constraint: Sender can only be used once (it's consumed by send()), and Receiver yields exactly one value. You can't accidentally send twice or forget to send — the compiler catches both.

This is the async equivalent of a promise/future in other languages. One task produces a value, another task awaits it. Clean, zero-overhead when the value is available, and integrates naturally with the async runtime.

The solution

use tokio::sync::oneshot;

async fn compute_answer() -> u64 {
    // Simulate expensive async work
    tokio::time::sleep(std::time::Duration::from_millis(100)).await;
    42
}

#[tokio::main]
async fn main() {
    let (tx, rx) = oneshot::channel();

    // Producer task
    tokio::spawn(async move {
        let answer = compute_answer().await;
        // Send the result (tx is consumed here — can only send once)
        let _ = tx.send(answer);
    });

    // Consumer awaits the result — no polling, no busy-waiting
    match rx.await {
        Ok(answer) => println!("answer: {}", answer),
        Err(_) => println!("producer dropped without sending"),
    }
}

Why the other options fail

B. mpsc channel — designed for multiple producers and a stream of messages. For a single one-time result, it's over-specified. It allocates a buffer, supports multiple sends, and the receiver is a stream, not a single future. oneshot models the constraint exactly.

C. Mutex with polling loop — busy-waiting burns CPU cycles checking the lock every 10ms. It doesn't integrate with the async runtime — the task isn't suspended, it's actively looping. This defeats the purpose of async.

D. std::sync::Condvar — a synchronous primitive. Calling condvar.wait() blocks the thread, starving other async tasks. It doesn't yield to the async executor. Use tokio::sync primitives in async code.

One producer, one consumer, one value: that's a oneshot channel.
