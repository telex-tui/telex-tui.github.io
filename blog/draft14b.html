Two components need to mutate shared config. No threads — Solution

March 2026 · Part B: The Solution

The answer: B

Use Rc<RefCell<Config>>.

Rc gives you shared ownership — both Renderer and PluginManager hold a handle to the same config, and the config lives until both are dropped. RefCell moves the borrow checking to runtime — you call .borrow() for read access and .borrow_mut() for write access. If you violate the rules (e.g., borrow mutably while someone else is borrowing), it panics instead of refusing to compile.

This is the standard single-threaded shared-mutable-state pattern in Rust. The compiler can't prove at compile time that the borrows don't overlap, because both components could theoretically access the config at the same time. RefCell defers that check to runtime, where it can see that the borrows actually happen at different times.

It's a trade-off: you lose compile-time borrow checking for flexibility. But in single-threaded code, the runtime cost is close to zero — it's just a counter bump.

The solution

use std::cell::RefCell;
use std::rc::Rc;

struct Config {
    theme: String,
    font_size: u32,
}

struct Renderer {
    config: Rc<RefCell<Config>>,
}

struct PluginManager {
    config: Rc<RefCell<Config>>,
}

impl Renderer {
    fn draw(&self) {
        let cfg = self.config.borrow();
        println!("Drawing with theme: {}, size: {}", cfg.theme, cfg.font_size);
    }
}

impl PluginManager {
    fn apply_plugin_settings(&self) {
        // borrow_mut gives us write access through a shared reference
        self.config.borrow_mut().font_size = 16;
    }
}

fn main() {
    let config = Rc::new(RefCell::new(Config {
        theme: "dark".into(),
        font_size: 14,
    }));

    let renderer = Renderer { config: Rc::clone(&config) };
    let plugins = PluginManager { config: Rc::clone(&config) };

    plugins.apply_plugin_settings();
    renderer.draw(); // sees font_size = 16
}

Why the other options fail

A. &mut Config for both — Rust's fundamental rule: you can't have two mutable references to the same data at the same time. The compiler will reject this.

C. Arc<Mutex<>> — works, but Mutex has locking overhead designed for thread safety. In single-threaded code, RefCell does the same job with a simple counter instead of an OS-level lock.

D. Raw pointer — throws away all of Rust's safety guarantees. You'd need unsafe for every access, and you'd be responsible for ensuring no aliasing violations. There's no reason to do this when RefCell exists.

Rc for shared ownership, RefCell for shared mutation — that's the single-threaded pair.
