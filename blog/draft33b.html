This blocking call is inside an async function. Fix it — Solution

March 2026 · Part B: The Solution

The answer: A

Use tokio::task::spawn_blocking.

Async runtimes like tokio use a small number of threads (usually one per CPU core) to run many tasks cooperatively. When a task calls a blocking function — one that takes milliseconds or more without yielding — it holds that thread hostage. No other task on that thread can make progress.

spawn_blocking moves the blocking work to a separate thread pool designed for exactly this. The async task awaits the result without blocking the executor. The blocking thread does the I/O. When it's done, the result is sent back to the async task.

The alternative is tokio::fs::read_to_string, which internally does the same thing — it uses spawn_blocking under the hood. Either approach works. The principle is the same: keep blocking work off the async executor threads.

The solution

use std::fs;

async fn load_template(path: String) -> String {
    // Move the blocking I/O to a thread that's allowed to block
    tokio::task::spawn_blocking(move || {
        fs::read_to_string(&path).unwrap_or_else(|_| "default".into())
    })
    .await
    .unwrap()
}

#[tokio::main]
async fn main() {
    let content = load_template("template.html".into()).await;
    println!("loaded: {} bytes", content.len());
}

Why the other options fail

B. Wrap in async {} — putting blocking code inside an async {} block doesn't make it non-blocking. The code still runs synchronously when polled. async is about structure, not magic — it doesn't change how the inner code executes.

C. .await the blocking call — std::fs::read_to_string doesn't return a Future. You can't .await it. It's a regular synchronous function. .await only works on types that implement Future.

D. std::thread::spawn with polling — std::thread::spawn returns a JoinHandle, not a Future. You can't .await it. You'd have to poll in a loop, which is busy-waiting — defeating the purpose of async. spawn_blocking returns a future that integrates properly with the async runtime.

Blocking code in async context: move it to spawn_blocking so the executor stays free.
