The caller shouldn't know what concrete type this is — Solution

March 2026 · Part B: The Solution

The answer: A

Return impl Database.

impl Trait in return position means: "I'm returning a specific concrete type, but I'm not telling you which one. All you know is that it implements this trait." The caller can call .query() but can't name PostgresDb or access Postgres-specific methods.

This gives you static dispatch — no heap allocation, no vtable. The compiler knows the concrete type and can inline method calls. The abstraction is purely at the API level, not at the runtime level.

Use impl Trait when the function always returns the same concrete type. If the function might return different types based on runtime logic (e.g., Postgres in production, SQLite in tests), you need Box<dyn Trait> instead. But when there's only one possible return type, impl Trait is the cleaner choice.

The solution

trait Database {
    fn query(&self, sql: &str) -> Vec<String>;
}

struct PostgresDb {
    connection_string: String,
}

impl PostgresDb {
    fn new(conn: &str) -> Self {
        PostgresDb { connection_string: conn.into() }
    }
}

impl Database for PostgresDb {
    fn query(&self, sql: &str) -> Vec<String> {
        vec![format!("result from: {}", sql)]
    }
}

// Caller sees: "something that implements Database"
// Caller cannot see: it's PostgresDb
fn create_db() -> impl Database {
    PostgresDb::new("host=localhost dbname=app")
}

fn main() {
    let db = create_db();
    let results = db.query("SELECT * FROM users");
    for r in results {
        println!("{}", r);
    }
}

Why the other options fail

B. Box<dyn Database> — works, but adds a heap allocation and virtual dispatch that aren't needed here. Use dyn when you need runtime polymorphism (different concrete types at different call sites). For a single concrete type, impl Trait is strictly better.

C. Generic parameter on function — with generics, the caller chooses the type: create_db::<PostgresDb>(). But the whole point is that the callee chooses. Generics flow from caller to callee; impl Trait flows from callee to caller.

D. Return trait directly — -> Database doesn't compile. Traits aren't sized — the compiler doesn't know how many bytes to allocate on the stack. You need either impl Trait (compiler knows the size) or Box<dyn Trait> (pointer with known size).

impl Trait in return position: the callee picks the type, the caller sees only the interface.
