This works single-threaded. Now make it work across threads — Solution

March 2026 · Part B: The Solution

The answer: B

Replace Rc with Arc and RefCell with Mutex.

Both pieces need to change. Rc uses non-atomic reference counting — incrementing and decrementing the counter from multiple threads is a data race. Arc uses atomic operations for the counter, making it safe to share across threads.

RefCell does runtime borrow checking with a simple cell — no synchronisation. If two threads call borrow_mut() at the same time, you get undefined behaviour (or a panic if you're lucky). Mutex replaces this with OS-level locking: one thread locks, does its work, unlocks. The others wait.

The mapping is mechanical: Rc → Arc, RefCell → Mutex, .borrow() → .lock().unwrap(), .borrow_mut() → .lock().unwrap(). Rust makes the single-threaded to multi-threaded transition explicit because the safety guarantees are fundamentally different.

The solution

use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let todos = Arc::new(Mutex::new(vec![
        String::from("buy milk"),
        String::from("write code"),
    ]));

    let writer = Arc::clone(&todos);
    let reader = Arc::clone(&todos);

    let write_handle = thread::spawn(move || {
        writer.lock().unwrap().push("read a book".into());
    });

    // Wait for writer to finish before reading
    write_handle.join().unwrap();

    let read_handle = thread::spawn(move || {
        let list = reader.lock().unwrap();
        for item in list.iter() {
            println!("- {}", item);
        }
    });

    read_handle.join().unwrap();
}

Why the other options fail

A. Arc with RefCell — RefCell is not Sync, so Arc<RefCell<T>> is not Send. The compiler will refuse to move it into a thread. RefCell's borrow tracking isn't thread-safe.

C. unsafe impl Send — lying to the compiler about thread safety. Rc's reference count is non-atomic. If two threads increment it concurrently, you corrupt the count and get use-after-free. This is exactly the kind of bug Send/Sync exist to prevent.

D. Add Send bounds — adding bounds doesn't make a type Send. Rc and RefCell don't implement Send/Sync, and bounds only constrain — they don't change the underlying type.

Rc→Arc and RefCell→Mutex: that's the mechanical translation from single-threaded to multi-threaded.
