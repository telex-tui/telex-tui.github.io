as_ / to_ / into_

This API takes an owned String. You have a &str. Cheapest path — Solution

March 2026 · Part B: The Solution

The answer: A

Use message.into_bytes().

into_bytes() consumes the String and returns its internal buffer as a Vec<u8>. Zero copy — the heap allocation that backed the String is now the Vec. No new memory allocated, no bytes copied. The into_ prefix tells you: the original is consumed, but the transformation is efficient because it reuses the existing data.

The original code borrows with as_bytes() (free, but produces &[u8]), then copies with to_vec() (allocates a new Vec and copies all bytes). Two steps, one wasted allocation. into_bytes() does it in one step with zero copies. When you don't need the original, into_ is almost always cheaper than as_ + to_.

The solution

fn send_packet(data: Vec<u8>) {
    println!("sending {} bytes", data.len());
}

fn main() {
    let message = String::from("PING");
    send_packet(message.into_bytes());  // zero-copy: String's buffer becomes the Vec
    // message is consumed — can't use it here
}

Why the other options fail

B. Keep as_bytes().to_vec() — it works, but it's wasteful. as_bytes() borrows for free, then to_vec() allocates and copies. Since message is never used again, the borrow-then-copy approach pays for a copy that into_ avoids entirely.

C. Change the API — changing send_packet to take &[u8] sidesteps the question. Sometimes APIs need owned data (async sends, thread transfers, buffering). Work with the API you have.

D. to_bytes() — to_bytes() doesn't exist on String. Even if it did, the to_ prefix means "create a new copy, leave the original." That would allocate — the same cost as as_bytes().to_vec(). When you don't need the original, into_ beats to_ by reusing the buffer.

The cost ladder: as_ borrows (free). to_ copies (allocates). into_ moves (reuses). When the original is done, skip straight to into_.
