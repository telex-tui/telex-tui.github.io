Concurrency Primitives: What Rust Inherited from C

March 2026

Draft — work in progress, not for publication.

If you learned concurrency through Rust's std::sync, you got clean APIs and compile-time guarantees. What you might not have got is a feel for why the primitives are shaped the way they are. That comes from the C level, where you're closer to the OS and the choices are more exposed.

This post maps C's concurrency primitives to their Rust equivalents — not to teach C, but to show where Rust's abstractions come from and what they're doing underneath. If you've read The GIL Is Just a Mutex, this is the companion piece: the foundations that make that post click.

Four groups. Each one a different answer to the same question: how do threads share work without corrupting each other's state?

1. Mutual Exclusion

The simplest concurrency problem: two threads must not be in the same critical section at the same time.

Mutex

One thread holds the lock. Other threads trying to acquire it will sleep until it's released. The holder is the only one who can release it. 

Time ──────────────────────────────────────────────────►

Thread A   ┌──lock()──┐                 ┌──lock()──┐
           │ critical  │                 │ critical  │
           │ section   │                 │ section   │
           └──unlock()─┘                 └──unlock()─┘

Thread B         ▓▓▓▓▓▓▓┌──lock()──┐         ▓▓▓▓▓▓▓
                 blocked │ critical  │         blocked
                         │ section   │
                         └──unlock()─┘

           ▓ = sleeping, waiting for lock

In C, this is pthread_mutex_lock / pthread_mutex_unlock. The OS deschedules the blocked thread — it doesn't burn CPU.

In Rust, it's std::sync::Mutex<T>. The critical difference from C: Rust's mutex owns the data it protects. You can't access T without holding the lock, because lock() returns a MutexGuard<T> that derefs to T. The guard drops at scope exit, releasing the lock automatically. 

// C: the link between mutex and data is a comment
pthread_mutex_t lock;
int shared_counter;  // protected by 'lock' (hope you read the comment)

// Rust: the link is the type system
let counter: Mutex<i32> = Mutex::new(0);
// Only way to reach the i32 is through lock()
*counter.lock().unwrap() += 1; // Guard drops here — lock released

This is the pattern across all four groups: Rust takes a C convention ("this mutex protects this data") and makes it a compiler-enforced invariant.

RwLock

A refinement of mutex. Many readers can hold the lock simultaneously, but a writer needs exclusive access. 

Time ──────────────────────────────────────────────────►

Reader A   ┌──read()───────────────┐
           │ reading                │
           └────────────────────────┘

Reader B      ┌──read()────────────┐
              │ reading (concurrent)│
              └─────────────────────┘

Writer C            ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓┌──write()──┐
                    blocked until     │ exclusive  │
                    all readers done  │ access     │
                                     └────────────┘

Reader D            ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓┌──read()──┐
                    blocked behind writer           │ reading  │
                                                    └──────────┘

In C: pthread_rwlock_rdlock / pthread_rwlock_wrlock. You choose read or write at each call site. Nothing stops you calling wrlock when you only needed to read.

In Rust: RwLock<T> gives you read() returning RwLockReadGuard (derefs to &T) and write() returning RwLockWriteGuard (derefs to &mut T). The type system ensures you can't mutate through a read guard — the borrow checker's & vs &mut distinction maps directly onto the shared/exclusive lock semantics.

Spinlock

Like a mutex, but the blocked thread doesn't sleep — it loops, burning CPU, checking repeatedly. 

Time ──────────────────────────────────────────────────►

Thread A   ┌──────────────────┐
           │ critical section  │
           └──────────────────┘

Thread B   ↻ ↻ ↻ ↻ ↻ ↻ ↻ ↻ ↻ ┌──────────────────┐
           spinning (burning    │ critical section  │
           CPU each iteration)  └──────────────────┘

           ↻ = one spin iteration (CPU busy, not sleeping)

The tradeoff: no syscall overhead to sleep and wake, but you're wasting cycles. Spinlocks win when the critical section is very short (nanoseconds) and contention is low. They lose badly if the holder gets preempted — the spinner burns an entire timeslice waiting for a thread that isn't even running.

In C: you'd typically build one from atomics (while (!atomic_compare_exchange(&locked, 0, 1)) {}). The kernel uses spinlocks heavily because it can disable preemption; in userspace, they're riskier.

Rust's standard library doesn't provide a spinlock, and that's deliberate. In userspace, an OS mutex is almost always the right choice — the kernel is smarter about scheduling than a busy loop. You'll find spinlocks in no_std crates and embedded work where there's no OS to defer to.

Recursive Mutex

A mutex that the same thread can lock multiple times without deadlocking. An internal counter tracks the depth; the lock is only truly released when the count hits zero. 

Time ──────────────────────────────────────────────────►

Thread A   ┌──lock() depth=1──────────────────────────┐
           │  ┌──lock() depth=2──────────────┐        │
           │  │  ┌──lock() depth=3──┐        │        │
           │  │  │  inner work      │        │        │
           │  │  └──unlock() depth=2┘        │        │
           │  └──unlock() depth=1────────────┘        │
           └──unlock() depth=0 (actually released)────┘

Thread B   ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓┌──lock()──
           blocked the entire time, all three depths

In C: pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_RECURSIVE). Common in large codebases where function A holds a lock and calls function B which also tries to acquire the same lock.

Rust doesn't provide a recursive mutex in std, and this is also deliberate. Recursive locking is usually a code smell — it means your lock boundaries are unclear. Rust's ownership model pushes you to restructure: separate the data access from the locking, pass &mut T into the inner function instead of re-locking. The borrow checker makes the cleaner design the path of least resistance.

2. Signalling

Mutual exclusion answers "who can access this now?" Signalling answers a different question: "how does one thread tell another that something happened?"

Condition Variable

A condvar lets a thread sleep until another thread signals that some condition has changed. Always used in combination with a mutex — the mutex protects the shared state, the condvar handles the waiting. 

Time ──────────────────────────────────────────────────►

Consumer   ┌──lock()──┐
           │ check:    │
           │ queue     │
           │ empty?    │
           │ yes →     │
           │ wait()────┘─── sleeping ──────────┌──woken──┐
           │ (mutex                             │ check:  │
           │  released                          │ queue   │
           │  while                             │ has     │
           │  sleeping)                         │ item →  │
           │                                    │ take()  │
           │                                    └──unlock()

Producer            ┌──lock()──┐
                    │ push()   │
                    │ signal() │──── wake one ──►
                    └──unlock()┘

The critical subtlety: wait() atomically releases the mutex and puts the thread to sleep. Without that atomicity, there's a gap where the producer could signal after the consumer checks but before it sleeps — the signal is lost, the consumer sleeps forever. This is the "lost wakeup" problem, and it's the entire reason condvars exist as a primitive rather than being something you build from a mutex and a sleep call.

In C: pthread_cond_wait(&cond, &mutex) / pthread_cond_signal(&cond). The pairing with a mutex is explicit but unenforced — you can pass the wrong mutex and get silent bugs.

In Rust: Condvar::wait() takes a MutexGuard, ensuring the condvar is always paired with the right lock. The type system prevents the wrong-mutex bug entirely. 

// C: hope you pair the right mutex with the right condvar
pthread_cond_wait(&cond, &mutex);  // wrong mutex? silent UB

// Rust: the guard IS the proof you hold the right lock
let guard = mutex.lock().unwrap();
let guard = condvar.wait(guard).unwrap(); // Can't pass a guard from a different mutex — types won't match

The other gotcha: spurious wakeups. Both POSIX and Rust condvars can wake without a signal. You must always recheck the condition in a loop. Rust's wait_while method handles this: 

let guard = condvar.wait_while(guard, |queue| queue.is_empty()).unwrap();
// Only proceeds when the queue is genuinely non-empty

Semaphore

A counter. wait() (or P/acquire) decrements it; if the count is already zero, the thread sleeps. post() (or V/release) increments it, waking a sleeper if any. 

Time ──────────────────────────────────────────────────►
                         Semaphore count: 2

Thread A   ┌─acquire()─────────────────┐
           │ count: 2→1                │
           │ working with resource     │
           └─release()─────────────────┘
                                        count: 1→2

Thread B      ┌─acquire()──────────────────────┐
              │ count: 1→0                      │
              │ working with resource            │
              └─release()───────────────────────┘
                                                 count: 0→1

Thread C         ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓┌─acquire()──────┐
                 count=0, sleeping     │ count: 1→0     │
                 until A releases      │ working        │
                                       └─release()──────┘

The key difference from a mutex: no ownership. Any thread can release a semaphore, not just the one that acquired it. And the count can be higher than 1 — "up to N threads can be in here." This makes it the right tool for resource pools (e.g., limiting to 10 concurrent database connections).

A binary semaphore (count of 1) looks like a mutex but isn't — the lack of ownership means thread A can acquire and thread B can release. This is useful for signalling ("I'm done, you can go") but dangerous for mutual exclusion (no guarantee the releaser was the holder).

In C: sem_wait / sem_post. POSIX semaphores are separate from pthread_mutex — different API, different semantics.

In Rust: std::sync doesn't include a semaphore. You'll find tokio::sync::Semaphore for async code. The standard library's position is that most semaphore use cases are better served by mutexes (for exclusion) or channels (for signalling). When you genuinely need "limit N concurrent accessors," the async semaphore is the idiomatic choice.

3. Atomic Operations

Locks protect critical sections. Atomics protect individual values — a single read-modify-write that the hardware guarantees is indivisible.

Compare-and-Swap (CAS)

The foundation of all lock-free programming. "If the value is still what I think it is, swap it to a new value. Otherwise, tell me what it changed to." 

Time ──────────────────────────────────────────────────►
                      Shared atomic: 42

Thread A   read: 42
           compute: 42 + 1 = 43
           CAS(expected=42, new=43) ──► succeeds ✓
                                        value is now 43

Thread B   read: 42
           compute: 42 + 1 = 43
           CAS(expected=42, new=43) ──► fails ✗
                                        value is 43, not 42
           retry: read 43
           compute: 43 + 1 = 44
           CAS(expected=43, new=44) ──► succeeds ✓
                                        value is now 44

No lock, no sleeping. The "loser" just retries. This works well when contention is low — most CAS attempts succeed first try. Under high contention, the retry loop can waste cycles, and you're back to needing a real lock.

In C: atomic_compare_exchange_strong(&val, &expected, desired). You also choose a memory ordering — memory_order_relaxed, memory_order_acquire, memory_order_seq_cst, etc. Getting the ordering wrong doesn't crash; it causes subtle, architecture-dependent, near-impossible-to-reproduce bugs.

In Rust: AtomicI64::compare_exchange(current, new, success_ordering, failure_ordering). The same memory orderings exist (Ordering::SeqCst, Ordering::Relaxed, etc.) and carry the same risks. Rust doesn't save you from choosing wrong here — this is one area where the type system can't encode the invariant, because the correctness depends on the relationship between multiple atomic operations across your whole program. 

use std::sync::atomic::{AtomicI64, Ordering};

let counter = AtomicI64::new(42);

// Simple atomic add — no CAS loop needed
counter.fetch_add(1, Ordering::SeqCst);

// CAS loop for complex updates
loop {
    let current = counter.load(Ordering::Acquire);
    let new_val = complex_computation(current);
    match counter.compare_exchange(
        current, new_val,
        Ordering::Release,  // on success
        Ordering::Relaxed,  // on failure
    ) {
        Ok(_) => break,     // swapped successfully
        Err(_) => continue, // someone else changed it, retry
    }
}

fetch_add, fetch_sub, fetch_or...

Hardware-level atomic arithmetic. Simpler than CAS — no loop, no retry, guaranteed to succeed in one shot. 

Time ──────────────────────────────────────────────────►
                      Shared atomic: 0

Thread A   fetch_add(1) ──► returns 0, value now 1
Thread B   fetch_add(1) ──► returns 1, value now 2
Thread A   fetch_add(1) ──► returns 2, value now 3

           Every increment is accounted for.
           No lost updates, no locks.

Use these when the operation is a single arithmetic step. The moment you need "read, compute something complex, write back," you're in CAS territory.

Memory Ordering

This is where atomics get genuinely hard. The question isn't whether an operation is atomic — it's about what other memory operations are visible to other threads before and after. 

          Thread A                    Thread B

          data = 42;
          flag.store(true, ???);      if flag.load(???) {
                                          // is data guaranteed
                                          // to be 42 here?
                                      }

The answer depends on the ordering:

Relaxed — no. The compiler and CPU can reorder. Thread B might see flag=true but data still at its old value.

Release (store) + Acquire (load) — yes. Release says "everything I wrote before this store is visible to anyone who acquires." Acquire says "I can see everything the releaser wrote." This is the pairing you want for most producer/consumer patterns.

SeqCst — yes, and all threads agree on a single total order of all SeqCst operations. Strongest guarantee, highest cost. The safe default when you're not sure.

In C, these are explicit: atomic_store_explicit(&flag, true, memory_order_release). In Rust, they're the Ordering enum. Both languages give you the same foot-gun. SeqCst everywhere is correct but slow. Weaker orderings are faster but demand you understand the happens-before relationships in your code.

The honest advice: use SeqCst until profiling shows it matters. Incorrect memory ordering creates bugs that don't show up on x86 (which has a strong memory model) but explode on ARM (which doesn't). You don't want that bug.

4. Thread Coordination

These primitives aren't about protecting data — they're about synchronising progress. "Everyone wait here until we're all ready."

Barrier

N threads arrive at a barrier. Each one blocks. When the Nth thread arrives, all are released simultaneously. 

Time ──────────────────────────────────────────────────►
                     Barrier(3)

Thread A   ──work──►│ wait ▓▓▓▓▓▓▓▓▓▓▓▓▓│──── continue ──►
                    │                     │
Thread B   ──work────────►│ wait ▓▓▓▓▓▓▓▓│──── continue ──►
                          │               │
Thread C   ──work──────────────►│all here!│──── continue ──►
                                │ RELEASE │

Classic use case: parallel computation in phases. All threads compute phase 1, barrier, all threads compute phase 2. Ensures no thread starts phase 2 while another is still on phase 1.

In C: pthread_barrier_init(&barrier, NULL, 3) / pthread_barrier_wait(&barrier).

In Rust: std::sync::Barrier::new(3) / barrier.wait(). The API is almost identical — barriers are simple enough that there's not much for the type system to add. The wait() returns a BarrierWaitResult that tells one (arbitrary) thread it's the "leader," useful if exactly one thread needs to do some post-barrier cleanup.

Once

Run initialisation exactly once, regardless of how many threads try simultaneously. The first caller runs the closure; all others block until it's done, then skip. 

Time ──────────────────────────────────────────────────►
                     Once::new()

Thread A   ──call_once()──►│ runs init │──► uses result ──►
                           │           │
Thread B   ──call_once()──►│ ▓▓▓▓▓▓▓▓▓ │──► uses result ──►
                           │ blocked   │
                           │ waiting   │
Thread C   ──────────────────call_once()──► skips, already done ──►

In C: pthread_once(&once_control, init_function). Or you see hand-rolled versions with double-checked locking — a mutex around the initialisation, guarded by a flag, with the flag check outside the lock for performance. Getting double-checked locking right in C requires careful memory ordering. Getting it wrong is a classic bug.

In Rust: std::sync::Once and, more commonly now, std::sync::OnceLock<T>. OnceLock combines the once-init with storage for the result — you initialise and store the value atomically. No hand-rolled double-checked locking needed. 

use std::sync::OnceLock;

static CONFIG: OnceLock<Config> = OnceLock::new();

fn get_config() -> &'static Config {
    CONFIG.get_or_init(|| {
        // Runs exactly once, even if 100 threads call simultaneously
        load_config_from_disk()
    })
}

This replaces a whole category of unsafe lazy-static patterns that C programmers would recognise: the global that needs initialising on first use, the flag that says whether it's been initialised, the mutex that protects the flag-check-and-init sequence.

The Pattern

Across all four groups, Rust does the same thing: it takes a C concurrency primitive and encodes its usage contract into the type system.

C convention (enforced by discipline) Rust enforcement
"This mutex protects this struct" Mutex<T> owns the data
"Only write-lockers should mutate" RwLockReadGuard gives &T, WriteGuard gives &mut T
"Always pair this condvar with this mutex" wait() takes a MutexGuard
"Check the condition after wakeup" wait_while() loops for you
"Release the lock when you're done" Guard drops at scope exit
"Initialise this global exactly once" OnceLock::get_or_init()

The primitives are the same. The operations are the same. The scheduling and performance characteristics are the same — Rust's Mutex calls pthread_mutex_lock underneath. What changes is that the rules a C programmer carries in their head become rules the compiler checks.

The one exception is memory ordering on atomics. There, Rust gives you the same Ordering enum as C's memory_order, and you can get it wrong just as easily. The type system can't help because the correctness depends on relationships between operations, not properties of a single value. This remains expert territory in both languages.
