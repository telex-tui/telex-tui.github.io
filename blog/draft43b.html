_map combiners

Filter a Vec of strings, parse the survivors, discard parse errors — Solution

March 2026 · Part B: The Solution

The answer: A

Use filter_map.

filter_map combines filter and map in a single pass. The closure returns Option<T>: Some(value) keeps the element (and transforms it), None discards it. One parse per element instead of two.

The _map suffix is the naming convention for combiners: the first word says what operation is combined with mapping. filter_map = filter + map. The pattern appears across the standard library — find_map, flat_map — always meaning "two operations in one pass."

The solution

fn main() {
    let inputs = vec!["42", "not_a_number", "7", "abc", "13"];

    let numbers: Vec<i32> = inputs.iter()
        .filter_map(|s| s.parse::<i32>().ok())  // parse once: Some(n) or None
        .collect();

    println!("{:?}", numbers);  // [42, 7, 13]
}

parse().ok() converts Result<i32, _> to Option<i32>. Successful parses become Some(n) and are kept. Failures become None and are discarded. One operation, one pass.

Why the other options fail

B. flat_map — flat_map maps each element to an iterator and flattens the results. You could make it work (parse().ok() returns an Option, which is iterable), but the intent is wrong. flat_map says "each element produces multiple items." filter_map says "each element produces zero or one item." Use the name that matches your intent.

C. Reorder to map then filter — map would produce Result values. Then filter would check them. Then you'd need another map to unwrap. Three steps, and the double-handling problem remains. filter_map is one step.

D. find_map — find_map stops at the first match. It returns a single Option<T>, not an iterator. You want all valid numbers, not just the first one.

When you need to test-and-transform in one pass, reach for filter_map. Return Some to keep, None to discard.
