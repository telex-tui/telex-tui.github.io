This type needs to be both printable and sortable — Solution

March 2026 · Part B: The Solution

The answer: A

Implement fmt::Display for printing. Derive (or implement) Ord for sorting.

Printing and sorting are separate concerns handled by separate traits. Display controls how a value looks when you write println!("{}", score). Ord controls how values compare when you call .sort(). There's no reason to couple them.

For sorting, you need four traits: PartialEq, Eq, PartialOrd, and Ord. For simple structs, you can derive all four. If you want custom ordering (e.g., sort by points descending), implement Ord manually and derive the rest.

Display can't be derived — you always implement it manually. That's by design: how a value is displayed to users is a deliberate choice, not something the compiler should guess.

The solution

use std::fmt;
use std::cmp::Ordering;

#[derive(Eq, PartialEq)]
struct Score {
    name: String,
    points: u32,
}

// Custom Display for user-facing output
impl fmt::Display for Score {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "{}: {} pts", self.name, self.points)
    }
}

// Sort by points descending (highest first)
impl Ord for Score {
    fn cmp(&self, other: &Self) -> Ordering {
        other.points.cmp(&self.points)
    }
}

impl PartialOrd for Score {
    fn partial_cmp(&self, other: &Self) -> Option<Ordering> {
        Some(self.cmp(other))
    }
}

fn main() {
    let mut scores = vec![
        Score { name: "Alice".into(), points: 150 },
        Score { name: "Bob".into(), points: 230 },
        Score { name: "Carol".into(), points: 180 },
    ];

    scores.sort();

    for (rank, score) in scores.iter().enumerate() {
        println!("#{}: {}", rank + 1, score);
    }
}

Why the other options fail

B. Implement ToString directly — the standard library says: "implement Display, and you get ToString for free." Implementing ToString directly bypasses Display, so println!("{}", score) won't work. And a custom compare method doesn't integrate with .sort() — the standard library expects Ord.

C. Debug for both — Debug output is for developers, not users. It prints things like Score { name: "Alice", points: 150 }. And you can't sort by debug output — that's comparing string representations instead of semantic values.

D. Single to_comparable_string method — conflates two independent concerns into one function. Sorting by string representation is fragile (is "9" greater than "10"?) and slow (allocating strings for every comparison). Keep display and ordering separate.

Separate concerns, separate traits: Display for humans, Ord for ordering.
