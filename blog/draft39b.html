_or vs _or_else

Three Options. Chain them so the first Some wins — Solution

March 2026 · Part B: The Solution

The answer: A

Use .or() to chain pre-computed Options, then .unwrap_or_else() for the final default.

Option::or() returns self if it's Some, otherwise returns the alternative. Chaining .or() creates a priority chain: first Some wins. Here both cli and env are already computed — they're Option<String> values sitting in local variables. No reason to wrap them in closures.

The final fallback calls default_value(), which allocates a String. That's work, so it goes behind unwrap_or_else. The chain reads like a sentence: "use CLI, or env, or else compute the default."

This is where the _or / _or_else distinction matters most: mix them in the same chain. Use .or() for values you already have, .or_else() for values that need computing.

The solution

fn from_cli() -> Option<String> {
    None
}

fn from_env() -> Option<String> {
    Some("from-env".into())
}

fn default_value() -> String {
    "fallback".into()
}

fn main() {
    let value = from_cli()
        .or(from_env())                       // already computed, use .or()
        .unwrap_or_else(|| default_value());   // needs work, use _or_else

    println!("resolved: {}", value);
}

Why the other options fail

B. or_else for everything — wrapping an already-computed Option in or_else(|| env) adds a closure for no reason. The value already exists. .or(env) is simpler and communicates that no computation is needed.

C. .or() for everything — using .or(Some(default_value())) eagerly allocates the default String even when the CLI or env value exists. This is the original bug wearing different clothes.

D. and_then — and_then chains operations when you have a value (Some → do more). Here you want the opposite: chain fallbacks when you don't have a value (None → try next). .or() is for fallback chains, .and_then() is for success chains.

Chain .or() for pre-computed alternatives, switch to _or_else when computation is needed.
