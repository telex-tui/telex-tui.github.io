Make this generic — but only for types that can fail — Solution

March 2026 · Part B: The Solution

The answer: A

Use a FromStr trait bound.

str::parse() already does exactly this — it's generic over any type that implements FromStr. The trait has an associated Err type, so each implementation defines its own error. Your generic function just needs to forward that constraint.

The bound T: FromStr says: "I accept any type that knows how to parse itself from a string, and my error type is whatever that type's parse error is." The compiler monomorphises this — for u16 it generates one version, for f64 another. Full type safety, zero runtime cost.

This is the core pattern of Rust generics: constrain, don't erase. You don't need to know the concrete type. You just need to know it has the capability you need.

The solution

use std::str::FromStr;
use std::fmt;

// Generic: works for any type that implements FromStr
fn parse_value<T: FromStr>(input: &str) -> Result<T, T::Err>
where
    T::Err: fmt::Debug,
{
    input.parse::<T>()
}

fn main() {
    let port: u16 = parse_value("8080").unwrap();
    let ratio: f64 = parse_value("3.14").unwrap();
    let flag: bool = parse_value("true").unwrap();

    println!("port={}, ratio={}, flag={}", port, ratio, flag);

    // This correctly fails at runtime
    let bad: Result<u16, _> = parse_value("not_a_number");
    println!("bad parse: {:?}", bad);
}

Why the other options fail

B. Box<dyn Any> with downcast — throws away compile-time type checking. The caller has to downcast at runtime and handle the case where the type is wrong. This is the Java Object pattern. Rust generics exist to avoid it.

C. Return impl Error — impl Error as a return type hides the concrete error type but doesn't help with the input generic. The challenge is about generalising the input type, not the error type.

D. Macro for each type — generates near-identical code for each type with no type-safety benefit over generics. Macros are a code generation tool, not an abstraction tool. Generics give you the same monomorphisation with better error messages and IDE support.

Trait bounds constrain generics to types with the right capabilities — that's the whole point of bounded generics.
